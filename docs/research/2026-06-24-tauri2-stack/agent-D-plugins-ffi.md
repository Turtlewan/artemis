## 1. Tauri 2 Plugin Model

### 1.1 Plugin vs. Inline Command — Decision Rule

**Use a plugin when:**
- The feature needs its own ACL permission namespace (e.g., `myplugin:allow-sign`).
- Mobile-specific implementations are required (Kotlin/Swift bridge via `mobile.rs`).
- The capability will be shared across multiple applications or released to the ecosystem.

**Use inline app commands when:**
- The feature is entirely app-specific, desktop-only, and has no reuse surface.
- You want to avoid the `build.rs` COMMANDS const + permissions scaffolding overhead.
- Caveat: Tauri 2 broke v1-style "inline plugins" — they require the same `build.rs` setup; they are not simpler. [[v2.tauri.app/develop/plugins/](https://v2.tauri.app/develop/plugins/), [discussion #9337](https://github.com/tauri-apps/tauri/discussions/9337)]

**For Artemis:** The crypto signing key commands (create, sign, destroy) should be a plugin (`tauri-plugin-artemis-keystore`) — they need a dedicated ACL permission set, have no mobile implementation needed initially (desktop-only first), and form a distinct security boundary.

### 1.2 Plugin Crate Structure

Bootstrap: `npx @tauri-apps/cli plugin new [name]` generates:
```
tauri-plugin-[name]/
├── src/
│   ├── commands.rs    # #[tauri::command] functions
│   ├── desktop.rs     # Desktop (Rust) implementation
│   ├── mobile.rs      # Mobile bridge (calls Kotlin/Swift)
│   ├── lib.rs         # Plugin init, Builder wiring
│   └── models.rs      # Shared types
├── permissions/       # TOML/JSON ACL definitions
├── android/           # Kotlin library (optional)
├── ios/               # Swift package (optional)
├── guest-js/          # TypeScript JS bindings (optional)
├── build.rs           # COMMANDS const → auto-gen permissions
└── Cargo.toml
```
[Source: [v2.tauri.app/develop/plugins](https://v2.tauri.app/develop/plugins/)]

### 1.3 Builder Pattern (`lib.rs`)

```rust
use tauri::{plugin::Builder, Runtime};

pub fn init<R: Runtime>() -> TauriPlugin<R> {
    Builder::<R>::new("artemis-keystore")
        .setup(|app, api| {
            // initialize state, open key store
            Ok(())
        })
        .invoke_handler(tauri::generate_handler![
            commands::create_key,
            commands::sign,
            commands::destroy_key,
        ])
        .build()
}
```

Lifecycle hooks available: `.setup()`, `.on_navigation()`, `.on_webview_ready()`, `.on_event()`, `.on_drop()`.

### 1.4 Exposing Commands

Commands use dependency injection; the plugin name prefixes the IPC call:

```rust
// src/commands.rs
#[tauri::command]
pub async fn sign<R: Runtime>(
    app: AppHandle<R>,
    payload: Vec<u8>,
) -> Result<Vec<u8>, String> { ... }
```

Frontend invocation:
```typescript
import { invoke } from '@tauri-apps/api/core';
await invoke('plugin:artemis-keystore|sign', { payload });
```

### 1.5 Permissions and ACL

**`build.rs` — autogeneration:**
```rust
const COMMANDS: &[&str] = &["create_key", "sign", "destroy_key"];
fn main() {
    tauri_plugin::Builder::new(COMMANDS).build();
}
```
This generates `allow-create-key`, `deny-create-key`, `allow-sign`, `deny-sign`, `allow-destroy-key`, `deny-destroy-key` in `permissions/autogenerated/commands/`.

**`permissions/default.toml`:**
```toml
"$schema" = "schemas/schema.json"

[default]
description = "Default Artemis keystore permissions — sign allowed, create/destroy require explicit grant"
permissions = ["allow-sign"]
```

**Capability file (`src-tauri/capabilities/default.json`):**
```json
{
  "identifier": "default",
  "windows": ["main"],
  "permissions": [
    "artemis-keystore:allow-sign",
    "artemis-keystore:allow-create-key"
  ]
}
```

**Critical:** Commands are blocked by default. Without an explicit `allow-*` entry in a capability, the frontend receives `Command X not allowed by ACL`. [[v2.tauri.app/learn/security/writing-plugin-permissions](https://v2.tauri.app/learn/security/writing-plugin-permissions/)]

### 1.6 Desktop vs. Mobile Split

`lib.rs` exports the platform-appropriate implementation:
- `desktop.rs`: pure Rust native code
- `mobile.rs`: bridge calling `PluginHandle::run_mobile_plugin("commandName", payload)`
- Android: Kotlin class extending `app.tauri.plugin.Plugin` with `@TauriPlugin` + `@Command` annotations
- iOS: Swift class extending `Plugin`, `@objc` methods

[Source: [v2.tauri.app/develop/plugins/develop-mobile](https://v2.tauri.app/develop/plugins/develop-mobile/)]

## 2. Native FFI — Windows CNG / NCrypt + TPM

### 2.1 Feasibility Verdict

**FEASIBLE.** The `windows` and `windows-sys` crates expose the full NCrypt/CNG API surface including `NCryptOpenStorageProvider`, `NCryptCreatePersistedKey`, `NCryptSetProperty`, `NCryptFinalizeKey`, and `NCryptSignHash`. P-256 (ECDSA) is supported by the Platform Crypto Provider via the `MS_PLATFORM_CRYPTO_PROVIDER` TPM backend.

### 2.2 Crates

```toml
# Prefer windows-sys for lower-overhead FFI; use windows for richer type safety
windows-sys = { version = "0.61", features = [
    "Win32_Security_Cryptography",
    "Win32_Foundation",
] }
# OR
windows = { version = "0.58", features = [
    "Win32_Security_Cryptography",
    "Win32_Foundation",
] }
```

`windows-sys` is lighter (no COM, no method wrappers) and sufficient for raw NCrypt calls.  
[Source: [microsoft.github.io/windows-docs-rs](https://microsoft.github.io/windows-docs-rs/doc/windows/Win32/Security/Cryptography/index.html), [dev.to TPM blog](https://dev.to/tsuruko12/how-i-used-tpm-for-key-encryption-in-rust-using-windows-apis-33n0)]

### 2.3 Key Creation Workflow (P-256 on TPM)

```rust
use windows_sys::Win32::Security::Cryptography::{
    NCryptOpenStorageProvider, NCryptCreatePersistedKey,
    NCryptSetProperty, NCryptFinalizeKey, NCryptSignHash,
    NCryptFreeObject,
    MS_PLATFORM_CRYPTO_PROVIDER,   // TPM provider
    NCRYPT_ECDSA_P256_ALGORITHM,   // = BCRYPT_ECDSA_P256_ALGORITHM = "ECDSA_P256"
    NCRYPT_UI_POLICY_PROPERTY,
    NCRYPT_UI_FORCE_HIGH_PROTECTION_FLAG,
    NCRYPT_PROV_HANDLE, NCRYPT_KEY_HANDLE,
};

// 1. Open TPM provider
let mut hprov: NCRYPT_PROV_HANDLE = 0;
NCryptOpenStorageProvider(&mut hprov, MS_PLATFORM_CRYPTO_PROVIDER, 0);

// 2. Create persisted P-256 key
let mut hkey: NCRYPT_KEY_HANDLE = 0;
let key_name: Vec<u16> = OsStr::new("ArtemisSigningKey").encode_wide()
    .chain(Some(0)).collect();
NCryptCreatePersistedKey(
    hprov,
    &mut hkey,
    NCRYPT_ECDSA_P256_ALGORITHM.as_ptr(),
    key_name.as_ptr(),
    0,  // dwLegacyKeySpec — must be 0 for CNG
    0,  // dwFlags
);

// 3. Set biometric/PIN gate BEFORE finalization
let ui_policy = NCRYPT_UI_POLICY {
    dwVersion: 1,
    dwFlags: NCRYPT_UI_FORCE_HIGH_PROTECTION_FLAG,  // 0x00000002
    pszCreationTitle: ptr::null(),
    pszFriendlyName: wide_str("Artemis Signing Key").as_ptr(),
    pszDescription: wide_str("Require PIN or biometric to sign").as_ptr(),
};
NCryptSetProperty(
    hkey,
    NCRYPT_UI_POLICY_PROPERTY.as_ptr(),
    &ui_policy as *const _ as *const u8,
    size_of::<NCRYPT_UI_POLICY>() as u32,
    0,
);

// 4. Finalize (after this, NCRYPT_UI_POLICY_PROPERTY is read-only)
NCryptFinalizeKey(hkey, 0);
```

**Key constraint:** `NCRYPT_UI_POLICY_PROPERTY` **must** be set before `NCryptFinalizeKey`. After finalization it is read-only.  
[Source: [learn.microsoft.com key-storage-property-identifiers](https://learn.microsoft.com/en-us/windows/win32/seccng/key-storage-property-identifiers), [NCRYPT_UI_POLICY struct](https://learn.microsoft.com/en-us/windows/win32/api/ncrypt/ns-ncrypt-ncrypt_ui_policy)]

### 2.4 NCRYPT_UI_POLICY Flags

| Flag | Value | Effect |
|------|-------|--------|
| `NCRYPT_UI_PROTECT_KEY_FLAG` | 0x00000001 | Show strong key UI as needed |
| `NCRYPT_UI_FORCE_HIGH_PROTECTION_FLAG` | 0x00000002 | Force high protection (PIN/biometric every use) |
| `NCRYPT_UI_APPCONTAINER_ACCESS_MEDIUM_FLAG` | 0x00000008 | Medium key accessed by app container |

Use `NCRYPT_UI_FORCE_HIGH_PROTECTION_FLAG` for the biometric/PIN gate.

### 2.5 Signing

```rust
// NCryptSignHash returns raw r||s (64 bytes for P-256)
let mut sig_len: u32 = 0;
NCryptSignHash(hkey, ptr::null_mut(), digest.as_ptr(), 32, ptr::null_mut(), 0, &mut sig_len, 0);
let mut sig = vec![0u8; sig_len as usize];
NCryptSignHash(hkey, ptr::null_mut(), digest.as_ptr(), 32, sig.as_mut_ptr(), sig_len, &mut sig_len, 0);
// sig is now 64 bytes: r[0..32] || s[32..64]
```

### 2.6 Windows Hello / KeyCredentialManager — RSA-Only Confirmation

`KeyCredentialManager` (Windows.Security.Credentials) creates **RSA 2048-bit keys** with PKCS #1 RSA PSS / SHA256 signatures. The API's Remarks section explicitly states: "The KeyCredentialManager class provides methods to manage key credentials, which are **RSA 2048-bit keys**." P-256 ECDSA is not available through this API.  
[Source: [learn.microsoft.com KeyCredentialManager](https://learn.microsoft.com/en-us/uwp/api/windows.security.credentials.keycredentialmanager?view=winrt-22621)]

**Use NCrypt + MS_PLATFORM_CRYPTO_PROVIDER directly for P-256 on TPM.**

## 3. Native FFI — macOS Secure Enclave

### 3.1 Feasibility Verdict

**FEASIBLE.** Two Rust approaches are available:

**Option A: `security-framework` crate (v3.7.0)** — idiomatic Rust wrappers around `Security.framework`. Covers `SecKey`, `SecAccessControl`, key generation/signing. Lighter binding layer.

**Option B: `objc2-security` crate** — generated bindings from `objc2` project covering the full Security framework ObjC API (`kSecAttrTokenIDSecureEnclave`, `SecKeyCreateSignature`, `SecAccessControlCreateWithFlags`, etc.). More verbose but exposes every API.

### 3.2 Key Creation (P-256, Secure Enclave, Touch ID gated)

The Secure Enclave requires:
- `kSecAttrKeyType = kSecAttrKeyTypeECSECPrimeRandom` (P-256)
- `kSecAttrTokenID = kSecAttrTokenIDSecureEnclave`
- `SecAccessControl` with `kSecAccessControlPrivateKeyUsage` + `kSecAccessControlBiometryAny` (Touch ID) or `kSecAccessControlUserPresence`

Pattern (via `security-framework` / `objc2-security`):
```rust
// Pseudo-code — actual call uses objc2 or security-framework wrappers
let access_control = SecAccessControl::create_with_flags(
    kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
    kSecAccessControlPrivateKeyUsage | kSecAccessControlBiometryAny,
)?;

let key_attrs = dict! {
    kSecAttrKeyType => kSecAttrKeyTypeECSECPrimeRandom,
    kSecAttrKeySizeInBits => 256,
    kSecAttrTokenID => kSecAttrTokenIDSecureEnclave,
    kSecPrivateKeyAttrs => dict! {
        kSecAttrIsPermanent => true,
        kSecAttrApplicationTag => b"dev.artemis.signing",
        kSecAttrAccessControl => access_control,
    },
};
let (public_key, private_key) = SecKey::generate_pair(key_attrs)?;
```

### 3.3 Signing (Touch ID gate, LAContext)

Use `kSecUseAuthenticationContext` with a pre-evaluated `LAContext` to supply the biometric result:

```rust
let la_context = LAContext::new();
la_context.evaluate_policy(
    LAPolicyDeviceOwnerAuthenticationWithBiometrics,
    "Artemis needs Touch ID to sign",
    callback,
)?;

let sign_attrs = dict! {
    kSecUseAuthenticationContext => la_context,
};
let signature: Vec<u8> = SecKey::create_signature(
    private_key,
    kSecKeyAlgorithmECDSASignatureDigestX962SHA256,  // returns DER
    digest,
    Some(sign_attrs),
)?;
```

[Source: [Apple Developer Forums](https://developer.apple.com/forums/thread/84309), [objc2-security crate](https://docs.rs/objc2-security)]

### 3.4 Crates for macOS

```toml
# Option A — higher-level
security-framework = { version = "3.7.0", features = ["OSX_10_15"] }

# Option B — lower-level, full API surface
objc2-security = "0.2"    # check current version on crates.io
objc2 = "0.5"
objc2-foundation = "0.2"
```

NEEDS-DOMAIN: `docs.rs/objc2-security` — accessible but content truncated in fetch; verify `kSecAttrTokenIDSecureEnclave` constant availability in the version you pin.

## 4. Signature Encoding — Normalization Required

| Platform | API | Output Format | Bytes (P-256) |
|----------|-----|---------------|---------------|
| Windows (NCrypt) | `NCryptSignHash` | Raw `r ‖ s` (big-endian scalars) | 64 fixed |
| macOS (SecKey) | `SecKeyCreateSignature` with `kSecKeyAlgorithmECDSASignatureDigestX962SHA256` | ASN.1 DER (X9.62) | 70–72 variable |

**Normalization path** (using the `ecdsa` crate, `p256` feature):

```rust
// Windows raw → DER (for a verifier that expects DER)
use ecdsa::Signature;
let raw_sig = Signature::<p256::NistP256>::from_bytes(&ncrypt_output_64)?;
let der_sig = raw_sig.to_der();

// macOS DER → raw (for a verifier that expects raw)
let raw_sig = Signature::<p256::NistP256>::from_der(&macos_der_output)?;
let raw_bytes = raw_sig.to_bytes();  // 64 bytes r||s
```

**Recommendation:** Normalise to DER at the plugin boundary before returning to the Tauri frontend. Store the canonical form consistently. The `ecdsa` crate (`Signature::from_bytes` / `from_der` / `to_der` / `to_bytes`) handles both paths.  
[Source: [docs.rs/ecdsa Signature](https://docs.rs/ecdsa/latest/ecdsa/struct.Signature.html), [Microsoft Learn NCryptSignHash](https://learn.microsoft.com/en-us/windows/win32/api/ncrypt/nf-ncrypt-ncryptsignhash), [sysadmins.lv ECC CNG](https://www.sysadmins.lv/blog-en/sign-data-with-ecc-cng-certificate.aspx)]

## 5. Common Tauri 2 Errors → Fixes

| # | Error / Symptom | Root Cause | Fix |
|---|-----------------|------------|-----|
| 1 | `Command X not allowed by ACL` | Command exists but no `allow-X` in capability | Add `<plugin-or-app-id>:allow-<command>` to `src-tauri/capabilities/default.json` |
| 2 | Plugin command silently blocked | Plugin added to `Cargo.toml` but no permissions granted | Add `<plugin>:default` or specific `allow-*` to capability |
| 3 | `unknown field 'allowlist'` on startup | `tauri.conf.json` is v1 format | Migrate: replace `allowlist` with capability files; rename fields (see #5, #6) |
| 4 | `devPath` or `distDir` not recognized | Field renamed in v2 | `devPath` → `devUrl`; `distDir` → `frontendDist` |
| 5 | `withGlobalTauri` not working | Config path changed | Moved from `build.withGlobalTauri` → `app.withGlobalTauri` [[#9248](https://github.com/tauri-apps/tauri/issues/9248)] |
| 6 | Blank window / app shows nothing | `frontendDist` path wrong or absolute | Use relative path from `src-tauri/`; for Vite: `"../dist"` |
| 7 | CSP blocks IPC calls | Custom CSP missing `tauri:` scheme | Add `"script-src": "'self' tauri:"` (and `default-src`) to CSP in `tauri.conf.json` |
| 8 | WebView2 not found (Windows) | WebView2 Runtime not installed for current user | Bundle WebView2 installer in `.msi` with `windowsInstaller.webviewInstallMode`; set to `embedBootstrapper` |
| 9 | `invoke is not a function` / invoke missing | `withGlobalTauri` not set; using old import path | Use `import { invoke } from '@tauri-apps/api/core'` (v2 path); or set `app.withGlobalTauri: true` |
| 10 | Android build: `NDK not found` | NDK env vars not set | Set `ANDROID_HOME`, `NDK_HOME`; install targets via `rustup target add aarch64-linux-android` |
| 11 | Linux build fails on WebKit | `libwebkit2gtk-4.1-dev` missing | `sudo apt install libwebkit2gtk-4.1-dev libgtk-3-dev libayatana-appindicator3-dev` |
| 12 | `IPC custom protocol failed` | Linux: webkit2gtk < 2.40 | Upgrade to webkit2gtk 2.40+ (required for `linux-protocol-body` IPC) [[#12835](https://github.com/tauri-apps/tauri/issues/12835)] |
| 13 | Capability `identifier` collision | Two capability files share same `identifier` | Each capability file needs a unique `identifier` field |
| 14 | Plugin not registered at runtime | `init()` not called in `tauri::Builder` | Add `.plugin(tauri_plugin_name::init())` to `tauri::Builder::default()` chain |
| 15 | `dialog.open not allowed` (or any official plugin) | Official plugin added to deps but permission not granted | Add e.g. `"dialog:default"` to capabilities permissions array |

[Sources: [v2.tauri.app/develop/plugins](https://v2.tauri.app/develop/plugins/), [fixdevs.com Tauri 2](https://fixdevs.com/blog/tauri-2-not-working/), [v2.tauri.app/security/capabilities](https://v2.tauri.app/security/capabilities/), [v2.tauri.app/learn/security/using-plugin-permissions](https://v2.tauri.app/learn/security/using-plugin-permissions/)]

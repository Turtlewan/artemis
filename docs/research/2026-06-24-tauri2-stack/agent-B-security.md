## 1. Capabilities & Permissions (ACL)

### 1.1 Mental Model

The v2 ACL is a **two-level system**: *Permissions* describe what a single command can do; *Capabilities* bundle permissions and bind them to one or more windows/webviews. [VERIFIED — v2.tauri.app/security/capabilities, /security/permissions]

```
Capability (JSON/TOML in src-tauri/capabilities/)
  ├── identifier         — stable name, used to activate in tauri.conf.json
  ├── windows            — ["main", "settings", …]  (which webviews get these)
  ├── webviews           — (optional, v2 mobile / multi-window)
  └── permissions[]
        ├── "plugin:permission-id"   — simple string reference
        └── { identifier, allow[], deny[] }  — scoped entry
```

**Default posture: deny everything.** No command in any plugin or app is callable from the frontend unless a capability file grants it. [VERIFIED — v2.tauri.app/security/permissions: "all potentially dangerous plugin commands and scopes are blocked by default"]

### 1.2 Capability File Structure

Location: `src-tauri/capabilities/<name>.json` (or `.toml`). All files in the directory are auto-loaded unless `tauri.conf.json` sets an explicit `capabilities` array — once that array is set, only listed capabilities are compiled in. [VERIFIED — v2.tauri.app/security/capabilities]

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "main-capability",
  "description": "Capability for the main window",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "fs:default",
    {
      "identifier": "fs:allow-read-text-file",
      "allow": [{ "path": "$APPDATA/myapp/**" }],
      "deny":  [{ "path": "$APPDATA/myapp/secrets/**" }]
    }
  ]
}
```

Core permissions use the `core:` prefix (e.g., `core:path:default`, `core:event:default`). Plugin permissions use `<plugin-name>:<permission-id>` (e.g., `fs:allow-read`, `dialog:open`). App-local commands use bare names. [VERIFIED — tauri-docs, migration guide]

### 1.3 How Plugins Expose Permissions

In the plugin's `build.rs`:
```rust
const COMMANDS: &[&str] = &["read_vault", "write_vault"];
fn main() {
    tauri_plugin::Builder::new(COMMANDS).build();
}
```
The build system auto-generates `allow-read-vault` / `deny-read-vault` (and analogues) into `permissions/autogenerated/commands/`. Plugins can additionally define hand-written TOML permissions with scopes for deeper control. [VERIFIED — v2.tauri.app/develop/plugins]

```toml
[[permission]]
identifier = "allow-read-vault"
description = "Allows reading from the vault"
commands.allow = ["read_vault"]

[[permission.scope.allow]]
path = "$APPDATA/vault/**"
```

### 1.4 Scopes

A **scope** is a typed constraint beyond command-level allow/deny. Scope objects are serde-deserialised into whatever type the plugin expects (`path`, `url`, `binary`, etc.). Both `allow` and `deny` lists are evaluated at runtime; deny wins. [VERIFIED — v2.tauri.app/security/permissions, plugin/file-system]

## 2. Content Security Policy (CSP)

### 2.1 Where to Configure

`tauri.conf.json` → `app.security.csp`. Value is either a string (`"default-src 'self'"`) or a directive-keyed object. [VERIFIED — v2.tauri.app/reference/config]

```json
{
  "app": {
    "security": {
      "csp": {
        "default-src": "'self'",
        "connect-src": "ipc: http://ipc.localhost",
        "script-src":  "'self'",
        "style-src":   "'self' 'unsafe-inline'",
        "img-src":     "'self' data: blob:",
        "font-src":    "'self'",
        "frame-src":   "'none'",
        "object-src":  "'none'",
        "base-uri":    "'none'"
      }
    }
  }
}
```

**Tauri injects nonces and hashes at compile time** into every HTML file, tightening `script-src` / `style-src` automatically. [VERIFIED — v2.tauri.app Isolation docs, GitHub issue #3831]

### 2.2 Custom Protocols

| Protocol | Purpose | Platform note |
|---|---|---|
| `tauri://localhost` | Serves bundled frontend assets | Linux / macOS |
| `https://tauri.localhost` | Serves bundled frontend assets | Windows (WebView2 requires HTTPS scheme) |
| `ipc:` / `http://ipc.localhost` | IPC postMessage bridge | All platforms — must be in `connect-src` |
| `asset:` / `http://asset.localhost` | Serves arbitrary disk files via `convertFileSrc()` | Disabled by default; requires `app.security.assetProtocol.enable: true` + scope |

[VERIFIED — v2.tauri.app/security/csp, /security/asset-protocol, GitHub issues #8476, #14422]

**Important:** `connect-src` must include **both** `ipc:` and `http://ipc.localhost` because the IPC protocol URI differs across webview engines. [COMMUNITY — tauri GitHub issue #12835, v2.tauri.app/reference/config example]

### 2.3 Strict CSP for a No-Remote-Content Client

For a privacy-first app that loads no remote assets (all bundled, no CDN):

```json
"csp": {
  "default-src":  "'self'",
  "connect-src":  "ipc: http://ipc.localhost",
  "script-src":   "'self'",
  "style-src":    "'self' 'unsafe-inline'",
  "img-src":      "'self' data: blob:",
  "font-src":     "'self'",
  "media-src":    "'none'",
  "frame-src":    "'none'",
  "object-src":   "'none'",
  "base-uri":     "'self'",
  "form-action":  "'none'"
}
```

Do NOT include `'unsafe-eval'`, external CDN origins, or `*`. Do NOT set `"csp": null`. [VERIFIED — v2.tauri.app/security/csp, official recommendation]

### 2.4 `dangerousDisableAssetCspModification`

This flag tells Tauri NOT to inject its compile-time nonces/hashes into the named directives. It exists to work around edge cases where the auto-injection breaks third-party loaders. **Never set this in a security-sensitive app.** Without it, Tauri's auto-injection is your defence-in-depth against injected scripts. [VERIFIED — v2.tauri.app/reference/config, GitHub issue #3831]

### 2.5 `app.security.headers`

Additional HTTP-like security headers injected by the asset server (separate from CSP):

```json
"headers": {
  "Cross-Origin-Opener-Policy":   "same-origin",
  "Cross-Origin-Embedder-Policy": "require-corp"
}
```

[VERIFIED — v2.tauri.app/security/http-headers]

### 2.6 Cross-Platform CSP Behaviour Notes

- **WebView2 (Windows):** Chromium-based, tracks Edge; CSP enforcement is generally strong. Assets served over `https://tauri.localhost` (required for custom-protocol origin model in Chromium). [VERIFIED — v2.tauri.app/reference/webview-versions]
- **WKWebView (macOS/iOS):** WebKit-based; served from `tauri://localhost`. CSP enforcement is strict and current as of Safari engine version shipped with OS. [VERIFIED — webview-versions ref]
- **WebKitGTK (Linux):** Same WebKit engine family, CSP support matches; older distros may ship older WebKitGTK versions with gaps. [COMMUNITY — dev.to/shrsv, wry GitHub repo]
- **Takeaway for security:** The IPC bridge URL differs per platform — always include both `ipc:` and `http://ipc.localhost` in `connect-src` or IPC will silently fail on one platform. Do not assume Chromium CSP semantics apply everywhere.

## 3. v1 → v2 Security Migration

### 3.1 Core Paradigm Shift

| Concept | Tauri v1 | Tauri v2 |
|---|---|---|
| Permission model | Flat `allowlist` in `tauri.conf.json` | ACL: `capabilities/` dir of JSON/TOML files |
| Granularity | Per-API-family toggle (fs, shell, dialog…) | Per-command, per-window, per-scope |
| Remote window support | Limited | First-class via `windows` array in capability |
| Default state | Allowlist fields off by default but additive | Deny-all; every capability is explicit |
| Frontend global | `window.__TAURI__` if `withGlobalTauri: true` | Same flag but moved: `build.withGlobalTauri` → `app.withGlobalTauri` |
| JS import path | `@tauri-apps/api/tauri` → `invoke` | `@tauri-apps/api/core` → `invoke` |
| IPC mechanism | `window.ipc.postMessage` | Custom protocol (`ipc:` / `http://ipc.localhost`) |

[VERIFIED — v2.tauri.app/start/migrate/from-tauri-1, blog/tauri-20]

### 3.2 Migration Steps

1. Install v2 CLI: `npm install @tauri-apps/cli@latest`
2. Run automated migration: `npx tauri migrate` — parses v1 `allowlist` and generates `src-tauri/capabilities/migrated.json`
3. Review generated capability file — migration tool maps allowlist families to v2 permissions but may over-grant; tighten scopes manually
4. Add `core:` prefix to any bare core permissions (e.g., `path:default` → `core:path:default`)
5. Update JS imports: `@tauri-apps/api/tauri` → `@tauri-apps/api/core`
6. Move `build.withGlobalTauri` → `app.withGlobalTauri` in `tauri.conf.json`

[VERIFIED — migration guide, blog/tauri-2-0-0-release-candidate]

### 3.3 Top Migration Gotchas

1. **Over-permissive migrated capability.** The auto-migrate tool generates a working config, not a minimal one. Review and drop unused permissions. [COMMUNITY — fixdevs.com/blog/tauri-2-not-working, pocketlantern.dev brief]
2. **`withGlobalTauri` location change.** If you relied on `window.__TAURI__` globally, the config key moved. Forgetting this = silent failure. [VERIFIED — migration guide]
3. **`core:` prefix required.** Post-RC, all core permissions need the `core:` prefix or the capability file silently drops them. [VERIFIED — tauri-docs beta→stable migration]
4. **IPC postMessage gone.** Code that called `window.ipc.postMessage` directly breaks. Use `@tauri-apps/api/core` invoke. [VERIFIED — migration guide]
5. **Plugin split.** `dialog`, `shell`, `notification`, `fs` are no longer part of core — they are separate crates (`tauri-plugin-dialog`, etc.) with their own permission identifiers. Add them as Cargo dependencies AND capability entries. [VERIFIED — v2.tauri.app/security/capabilities]
6. **Capability files are additive by default.** All files in `src-tauri/capabilities/` are compiled in unless you explicitly list them in `tauri.conf.json`. A stale test-capability file in the dir silently grants excess perms. [VERIFIED — capability docs]
7. **`dangerousRemoteUrlIpcAccess` is a sharp edge.** If you had remote URL IPC in v1, you must explicitly configure `security.dangerousRemoteUrlIpcAccess` in v2 — and you almost certainly should not for a local-only app. [VERIFIED — v2.tauri.app/blog/tauri-1-3]

## 4. Hardening Guidance — Privacy-First Local Client

### 4.1 Process Model

Tauri runs two processes: the **Core** (Rust, trusted, full OS access) and the **WebView** (untrusted, JS/TS, access only via IPC). The design principle is: push business logic and all secrets into Core; the WebView is a dumb render surface. [VERIFIED — v2.tauri.app/concept/process-model]

### 4.2 DevTools in Production

DevTools (web inspector) are **off by default in release builds**. They are on in debug builds. To turn them on in release (don't, for a prod privacy app):
```toml
# src-tauri/Cargo.toml — DON'T do this in prod
[dependencies]
tauri = { version = "2", features = ["devtools"] }
```
For a privacy-first client: verify the `devtools` feature is **absent** from `Cargo.toml` in your release profile. [VERIFIED — v2.tauri.app/develop/debug]

### 4.3 Isolation Pattern

The **Isolation Pattern** routes all IPC calls through a sandboxed `<iframe>` (your "isolation app") before they reach Tauri Core. It lets you validate, sanitise, or reject IPC messages in a second sandboxed JS context — useful when third-party JS could be loaded or when you want an extra gate on crypto commands.

```json
"app": {
  "security": {
    "pattern": {
      "use": "isolation",
      "options": { "dir": "../dist-isolation" }
    }
  }
}
```

For a closed local app with no third-party JS: isolation adds complexity for marginal gain. Recommended if you render any untrusted user content in the WebView. [VERIFIED — v2.tauri.app/concept/inter-process-communication/isolation]

### 4.4 Secrets — Never in the Frontend

The official docs state: **"never handle secrets in the Frontend."** [VERIFIED — v2.tauri.app/concept/process-model] Corollaries for a P-256 crypto auth client:
- Private keys, session tokens, tunnel credentials: store in Core (Rust), expose only derived artefacts (signatures, encrypted blobs) to the WebView via IPC.
- The `tauri-plugin-stronghold` (IOTA Stronghold, memory-safe encrypted store) is Tauri's recommended secret store for desktop; permissions must be explicitly granted. [VERIFIED — v2.tauri.app/plugin/stronghold]
- Never pass a raw private key as an IPC argument — IPC arguments are visible in WebView JS context.

### 4.5 Asset Protocol

`assetProtocol` is **disabled by default** (`enable: false`). Enable only if you need to serve local disk files to the WebView (e.g., user-selected media). Always pair with a tight scope:

```json
"assetProtocol": {
  "enable": true,
  "scope": ["$APPCACHE/renders/**"]
}
```

[VERIFIED — v2.tauri.app/security/asset-protocol]

### 4.6 What NOT to Expose to the WebView

- Raw filesystem commands beyond the minimum needed (use scoped `fs:allow-*` not `fs:default`)
- Shell execute (`shell:execute`) — exposes arbitrary process spawn; use `shell:allow-open` only if needed
- `http` plugin without URL scopes (scope to specific internal endpoints only)
- `window:create` without careful review — browser arguments were historically a privilege-escalation vector (CVE-adjacent, fixed by moving to Rust-side only) [VERIFIED — v2.tauri.app/blog/tauri-1-3]

## 5. WebView Security Notes (Cross-Platform, Security-Relevant)

| Engine | Platform | Security-relevant notes |
|---|---|---|
| WebView2 | Windows | Chromium-based, auto-updates with Edge; generally strong CSP; serves from `https://tauri.localhost` (not `tauri://`) — include this in any hardcoded origin checks |
| WKWebView | macOS/iOS | WebKit; serves from `tauri://localhost`; sandboxed at OS level; strict CSP |
| WebKitGTK | Linux | WebKit; older distro packages may lag on CSP features; no OS-level sandbox by default [COMMUNITY — wry repo, dev.to] |
| All | — | WebView memory-unsafety is acknowledged in Tauri's threat model as the largest risk surface; they plan future sandboxing work [VERIFIED — v2.tauri.app/security/future] |

**The platform protocol split** (`https://tauri.localhost` vs `tauri://localhost`) means:
- Do not hardcode either in Rust code or IPC handlers — use `tauri::api::path` helpers and the `AppHandle` for origin resolution.
- CSP `connect-src` must include both `ipc:` and `http://ipc.localhost` to cover all platforms. [VERIFIED + COMMUNITY — GitHub issues, reference/config]

## 6. NEVER / ALWAYS Rules for Tauri 2 Security

### NEVER
- **NEVER** expose a command in `capabilities/` that the app doesn't use. Every unused permission is attack surface. Audit after migration.
- **NEVER** set `"csp": null` in any environment — this disables all CSP protection entirely.
- **NEVER** use `dangerousDisableAssetCspModification` unless you have re-implemented nonce/hash injection yourself.
- **NEVER** store or pass secrets (private keys, tokens, passwords) as IPC arguments or in the JS frontend layer. IPC args are visible in the WebView JS scope.
- **NEVER** enable the `devtools` Cargo feature in a release build for a privacy app.
- **NEVER** enable `shell:execute` or any broad shell permission — scope to the minimum binary + args needed, or implement the shell call in Rust.
- **NEVER** enable `assetProtocol` without a tight path scope — an open asset protocol is an arbitrary file read from the web side.
- **NEVER** use `dangerousRemoteUrlIpcAccess` unless you fully own and control the remote domain; never for a local-only app.
- **NEVER** put `window.__TAURI__` (via `withGlobalTauri`) in production unless all call sites use it — it bloats the JS global and makes IPC enumeration trivial.
- **NEVER** leave test capability files in `src-tauri/capabilities/` in production builds — they are silently compiled in.
- **NEVER** assume a v1 `allowlist` → v2 auto-migrated capability is minimal — it will be over-permissive; review every entry.

### ALWAYS
- **ALWAYS** include `core:default` plus only the specific plugin permissions you need — do not use wildcard or meta-sets unless you have audited what they grant.
- **ALWAYS** add `connect-src: ipc: http://ipc.localhost` to your CSP — both forms are needed for cross-platform IPC.
- **ALWAYS** pin capabilities to specific window identifiers (`"windows": ["main"]`) — never use `"*"` without a documented reason.
- **ALWAYS** scope filesystem permissions to the narrowest path (`$APPDATA/myapp/**`, not `$HOME/**`).
- **ALWAYS** add `deny` rules for sensitive sub-paths when using recursive allows (e.g., deny `$APPDATA/myapp/keys/**` even inside a broader allow).
- **ALWAYS** handle secrets (crypto keys, tokens) exclusively in the Rust Core; expose only the result (signature, ciphertext) over IPC.
- **ALWAYS** verify the `devtools` Cargo feature is absent in release profiles (`cargo tree -e features | grep devtools`).
- **ALWAYS** set `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` in `app.security.headers` for isolation completeness.
- **ALWAYS** use `@tauri-apps/api/core` (not the v1 `@tauri-apps/api/tauri`) path for `invoke` — the old path is removed.
- **ALWAYS** use autogenerated `allow-<command>` / `deny-<command>` permissions from `build.rs` for custom plugins — hand-writing permission identifiers risks typos that silently deny.
- **ALWAYS** consider the Isolation Pattern if the WebView ever renders untrusted user content (HTML snippets, markdown from untrusted sources, etc.).

# Spec: M8-a — Google-auth foundation (single-owner OAuth2 installed-app flow, refresh-token in the owner-private encrypted store, auto-refreshing authorized-credentials factory, consent CLI)

**Identity:** The shared Google authentication foundation that the Calendar module + Gmail connector build on — a single-owner OAuth2 **installed-app (loopback) consent flow** (run as an owner-present CLI on the Mini), a `TokenStore` that persists the refresh token in the **owner-private M2 encrypted scope** (Tier-1, SQLCipher via the broker DEK), an auto-refreshing **authorized-credentials factory** the connectors call, and a scope registry connectors contribute to.
→ why: see docs/technical/adr/ADR-011-spoke-source-of-truth.md (Calendar/Gmail mirror domains; published-unverified single-owner OAuth, refresh token in the owner-private encrypted scope) · docs/technical/adr/ADR-005-owner-key-broker.md (the owner DEK + `KeyProvider` this persists under) · docs/technical/adr/ADR-009-untrusted-content-and-deep-research.md (controlled egress; secrets from Keychain/env, never committed) · the apex-google skill (installed-app flow, least-scope, refresh-token-server-side-encrypted, `sub`-not-email, `invalid_grant` handling).

## Assumptions
- **M0-a** (`Settings`/`get_settings`, `paths.scope_dir`, `pyproject.toml`, `.gitignore` ignoring `.env`), **M0-d** (`ports/` package, frozen-dataclass + `Protocol` conventions), **M2-b** (`KeyProvider` Protocol with `dek_for_scope`/`is_owner_unlocked`, `SecretKey`, `ScopeLockedError`, `OWNER_PRIVATE` constant from `artemis.identity.scope`), **M2-c** (`sqlcipher_open(path, key_hex)` raw-hex keyed-open wrapper in `artemis.data.sqlcipher`) are complete. → impact: Stop (M8-a extends these exact symbols; signatures must match).
- The refresh token is a **raw, high-value, long-lived secret** (it can read all of the owner's mail + calendar), therefore it is stored **Tier-1: owner-private, phone-unlock-required** — encrypted in a SQLCipher file opened with the owner DEK via the M2 broker. It is **never** placed in the Tier-0 always-on proactive corpus (ADR-006 bars raw secrets there; locked-state calendar awareness uses *derived* synced data, not the live token). → impact: Stop (this is the security boundary the M8-a security review checks).
- M8-a is **infrastructure, not a domain module** — it ships no `ModuleManifest`, no brain-facing tools, no proactive hooks. It is a library + a setup CLI consumed by the Calendar/Gmail specs. → impact: Caution (no manifest/registry wiring here; connectors register their own).
- The OAuth **app is "published / in production, unverified"** (single-owner): this avoids the 7-day refresh-token expiry that applies to apps in *Testing* publishing status, and accepts Google's "unverified app" consent warning (fine for one owner). The consent flow requests **`access_type=offline` + `prompt=consent` + `include_granted_scopes=true`** so a refresh token is reliably returned on every consent and incremental scope expansion accumulates grants. → impact: Caution (these exact OAuth params are required; without `prompt=consent` Google omits the refresh token on re-consent).
- M8-a is **scope-agnostic**: it hardcodes no Gmail/Calendar scope. Connectors register their scopes via the M8-a scope registry; the consent CLI grants the registered union (or an explicit `--scope` override). The wave-1 connectors (CAL, Gmail) pass their own least-privilege scopes in their own specs. → impact: Caution (M8-a defines the registry mechanism + ships it empty; it is not responsible for choosing connector scopes).
- The OAuth **client_id + client_secret** are read from the environment (`GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`), injected on the Mini from the macOS **Keychain** via the existing slot `.env`/launchd mechanism, **never committed** (`.gitignore` already ignores `.env`). For an installed app the client_secret is not truly confidential (PKCE is the real protection, enforced by the flow); it is still kept out of the repo and out of the per-scope token DB. → impact: Caution (mirrors the ADR-009 `BRAVE_API_KEY` secret pattern; the secret is config, not owner-private data — it is NOT stored in the SQLCipher token DB).
- Real Google network round-trips (consent, token exchange, refresh, an authorized API ping) and the real SQLCipher keyed persistence require real credentials + a browser + the broker + Apple Silicon → **GATED on-hardware** (Task 7), exactly as M2-c gates its real SQLCipher/mlock tasks. Off-hardware everything is tested behind fakes (`InMemoryTokenStore`, `FakeKeyProvider`, a stub OAuth flow). → impact: Stop (keeps M8-a CI-buildable off the Mini; the network/crypto reality is owner-present/on-hardware).
- The exact on-disk location of the token DB under the broker-mounted per-scope **`vault/`** (`/opt/artemis/<slot>/<scope>/vault/…` per ADR-007/M2-c) is reconciled at on-hardware integration (Task 7), identical to the M3-a LanceDB-path deferral. Off-hardware the path derives from `paths.scope_dir(settings, OWNER_PRIVATE)`; a one-line path adapter at integration repoints it under the mounted vault. → impact: Low (off-hardware tests don't care; one-line change on the Mini).

Simplicity check: considered storing the refresh token via the M4 `MemoryStore` / the M2-b memory `memory.db` — rejected: a credential is not a memory fact (different lifecycle, single-row, no bitemporal/decay), and M8-a must not couple to the unbuilt M4 schema; a tiny dedicated single-row SQLCipher store reusing only the `sqlcipher_open` primitive is the minimum. Considered the device-code or client-app-relayed consent flows — rejected at planning (owner chose the loopback installed-app flow). Considered hardcoding the Calendar+Gmail scopes here for one combined consent — rejected: incremental, connector-declared scopes are Google's recommended pattern and keep M8-a independent of the connector specs. The minimum is: a scope registry + a loopback consent flow + an owner-private token store + an auto-refreshing factory + a setup CLI.

## Specialist Context
### Security
M8-a holds the single most powerful owner credential in the system (a Google refresh token = standing read access to all mail + calendar). Invariants the build MUST honour:
- **Refresh token is owner-private Tier-1**, encrypted at rest in SQLCipher opened only with the broker-delivered owner DEK; it is NEVER written to the Tier-0 proactive corpus, a plaintext file, or a log (ADR-005/006).
- **Never logged, never `repr`'d**: `StoredToken` redacts `refresh_token`; `GoogleOAuthConfig` redacts `client_secret`; the CLI prints scopes/times, never tokens.
- **PKCE enforced** on the installed-app flow; `prompt=consent` + `access_type=offline` so the refresh token is the auth artifact; **least-privilege scopes** are the connectors' responsibility (M8-a is scope-agnostic; the registry sorts/dedupes).
- **`invalid_grant` is handled**, not swallowed: a revoked token surfaces as `ReauthRequiredError` driving the owner back through consent (apex-google checklist).
- **Client secret is config, not owner-data**: it lives in env/Keychain, never committed, and is NOT stored in the per-scope token DB (so it can rotate without touching owner data).
- Outbound calls hit only Google hosts (trusted-connector egress, ADR-009 allowlist); the consent loopback binds `127.0.0.1` only.
- **CSRF `state` on the consent callback** is a required invariant (`run_local_server` generates + validates a cryptographically-random `state`; a mismatched/forged callback `state` is rejected) — named explicitly, not relied on silently; on-hardware tamper-tested (Task 7e).
- **The authorized `Credentials` object must never be logged** by M8-a or its connector callers (`google.oauth2.credentials.Credentials.__repr__` is not a guaranteed redactor) — documented contract + a standing-guard repr test.
- **Loopback `redirect_uri` is supplied by `run_local_server`** (ephemeral port), never hardcoded — a static `redirect_uris` would break consent (`redirect_uri_mismatch`).
- **New deps pass `pip-audit`** at build (supply-chain gate).

### Performance
The credentials factory does ONE `refresh()` to mint a live access token, then returns the auto-refreshing `Credentials` so connectors reuse it across calls without re-reading the token DB each request; opening the SQLCipher token store is cheap and only happens on load/save, not per API call. The broker session window means `dek_for_scope(OWNER_PRIVATE)` does not re-hit the enclave per call (M2-c cache).

### Accessibility
(none — headless foundation; the consent UX is a one-time CLI in the owner session. The eventual in-app pairing/consent surface is a deferred client-app upgrade, not M8-a.)

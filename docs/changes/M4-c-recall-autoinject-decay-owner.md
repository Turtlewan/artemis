# Spec: M4-c — Auto-inject (current facts → system prompt, token-budgeted) wired into the M1 Brain turn loop + post-turn write-path scheduling + decay scoring (recency × salience × access, never-hard-delete) + owner view/edit/delete surface

**Identity:** Completes the memory engine on top of M4-a/M4-b: implements `inject_context` (rank current `as_of=now` facts above an inject threshold by a decay score — recency × salience × access — and pack into the system prompt within a token budget), wires auto-inject + the M4-b async write queue into the M1 `Brain.respond` turn loop (inject before the model call; schedule extraction after), implements the decay scorer (demote-below-threshold / tombstone, **never hard-delete**), and exposes the owner view/edit (= bitemporal UPDATE) / delete (tombstone) / explicit-purge surface with provenance. Schema/repository = M4-a; write-path decisioning = M4-b.
→ why: see docs/technical/adr/ADR-004-memory-engine.md (auto-inject each turn within a token budget; decay recency×salience×access, never-hard-delete, owner true-purge separate; owner view/edit=UPDATE/delete=tombstone with provenance) · docs/technical/architecture/brain.md § Memory (auto-inject; forgetting/decay; owner control) · docs/research/memory-engine-research.md (owner-edit human-in-the-loop; decay half-life tuning gated).

- M4-c needs the Brain to also hold a `MemoryStore` + a `MemoryWriteQueue`. M1-b's `Brain` is constructed with `(Router, ToolRegistry, ModelPort)`; M4-c ADDS optional `memory: MemoryStore | None = None` and `write_queue: MemoryWriteQueue | None = None` constructor params (default None → memory disabled, so M1/M2/M3 tests that build a `Brain` without memory still pass — backward-compatible). `compose_brain` is updated to build + inject them when an unlocked owner scope is available. → impact: Caution (additive, optional params keep prior specs' Brain construction valid; memory is only active for owner scopes with an open keyed DB).
- **Decay score** = `recency × salience × access` (ADR-004 Ebbinghaus-style). Drafting: `recency = exp(-Δt_days / half_life_days)` (Δt from `last_access` or `valid_from`); `salience` = the stored `salience` column (default 1.0; boosted for explicit owner-stated facts — set by M4-b confidence/owner edits); `access = 1 + log1p(access_count)`. `score = recency * salience * access * confidence`. Facts with `score < inject_threshold` are NOT injected (demoted); facts that decay far below are candidates for tombstone by a sweep — but **never hard-deleted**. → impact: Caution. [RESOLVED — drafted defaults, tuned at GATED Task 6: `half_life_days = 14.0`, `inject_threshold = 0.2`, `tombstone_floor = 0.02` as config-tunable constants; they do NOT change the code shape, only the constants. Weighting per ADR-004's composite forgetting score (see the composite-score note below); half_life_days/inject_threshold/tombstone_floor are placeholders tuned on traces at Task 6 (code-shape unchanged).]
- **Composite forgetting score (recall re-rank):** per ADR-004's brain-sweep absorption, recall on the SEMANTIC store applies the composite forgetting score `I(m,t)=α·exp(−λΔt)+β·access_count+γ·cosine(m,query)` as a retrieval-time re-rank MULTIPLIER (recently-accessed ×1.5, stale ×0.3) over sqlite-vec similarity — NOT eager deletion. The EPISODIC store stays TTL-only. Distinct from the inject-time decay score: inject ranks current facts by recency×salience×access; recall re-ranks vector candidates by the composite I(m,t) multiplier. → impact: Caution.
- **Auto-inject** (ADR-004): each turn, `inject_context(person_id, token_budget, as_of=now)` queries current (`as_of=now`) facts, ranks by decay score, filters above `inject_threshold`, and packs the top facts into a compact text block within `token_budget` (greedy by score; stop when the next fact would exceed the budget). The Brain prepends this block as a system message. Accessing a fact for injection BUMPS its `access_count` + `last_access` (so used facts decay slower — the access term). → impact: Stop (this is the auto-inject contract; the access-bump closes the recency×access loop).
- **Owner control** (ADR-004): the owner can view a fact's full history + provenance, edit it (= a bitemporal `repo.update` — a NEW version, auditable, the old version preserved), delete it (tombstone), or **explicitly purge** it (the ONLY hard-delete path — a separate, explicit owner action that removes the version rows + vec/fts rows for a `fact_key`, distinct from tombstone). Owner edit is **human-in-the-loop** (the research-required mitigation) — the surface returns the current fact + asks the owner to confirm the new value; no automatic owner-edit. → impact: Stop (view/edit/delete/purge with provenance; purge is the only hard-delete and is explicit; edit is human-in-the-loop).
- The owner surface is a **Python API on the memory package** (functions/class methods), NOT an HTTP/UI surface (no frontend in M4; the app surface is a later milestone). It is callable from a future owner UI / a CLI. → impact: Low (M4-c ships the callable surface + tests; the UI wiring is deferred).
- Off-hardware: everything is deterministic (FakeEmbedder, the M4-a DB fixture, a `FakeModelPort` for the Brain). **The decay half-life / inject-threshold tuning on real usage traces is the GATED Task 6.** → impact: Caution.

Simplicity check: considered a memory TOOL the model calls to fetch facts — rejected: ADR-004/brain.md lock AUTO-inject (the model never calls a memory tool; facts are packed into the system prompt each turn). Considered a background decay daemon — rejected for M4: decay is computed at inject time (lazy, no scheduler) + a simple on-demand `sweep_tombstone_candidates` the owner/maintenance can call; a scheduled sweep is a later concern. Considered making owner-edit automatic — rejected: research requires human-in-the-loop. This is the minimum that satisfies auto-inject + decay + owner-control.

## Specialist Context
### Security
Auto-inject only ever runs for an **owner scope with an open keyed DB** (the wall: a locked/guest scope has no memory store → no inject). The inject block contains sensitive owner facts — it goes ONLY into the LOCAL responder's system prompt (never to a cloud model; the sensitivity/escalation gate is a separate layer and the M1 escalation path is still a stub). NEVER log the inject block / fact objects at info level. **never-hard-delete** is preserved everywhere EXCEPT the explicit, confirm-gated owner `purge_fact` (the single, documented, irreversible hard-delete) — every other "delete" is a tombstone with history intact. Owner-edit is human-in-the-loop (confirm-gated). A memory-read failure degrades to "no inject" — it never breaks the turn or leaks. [FLAG for apex-security (M4 gate): confirm the inject block never reaches a cloud/escalation path; confirm `purge_fact` is the only hard-delete and is confirm-gated; confirm no fact plaintext is logged; review the inject-block prompt-injection surface (owner's own facts are trusted, but extracted facts originate from untrusted turn content — spotlight if a fact ever derives from ingested third-party content).]

### Performance
Decay is computed **lazily at inject/recall time** (no scheduler, no background daemon in M4) — O(current facts) per turn, which is small at per-person scale. Auto-inject runs once per turn before the model call; the access-bump is a single cheap UPDATE per injected fact. The token-budget greedy pack bounds the injected context. The write path is async (M4-b) — scheduling it after the turn adds no response latency. Half-life/threshold tuning (Task 6) is the only empirical knob.

### Accessibility
(none — the owner surface is a Python API; the eventual owner-memory UI is a later milestone and inherits apex-accessibility then)

# Spec: CLIENT-screens — domain detail content (Review · Status · the locked bespoke domain views + generic tails, filling the CLIENT-card overlay registry)

**Identity:** Implements the concrete `DomainDetail` React components that fill the CLIENT-card detail-overlay registry — **Review** (recipes + GATE pending-actions), **Status** (lock/connection UI), and the **per-domain detail views LOCKED in DECISIONS-LOG 2026-06-23** ("Expanded-card detail views"): bespoke **Calendar** (Month+Week+selected-day), **Tasks** (Due/Overdue/Upcoming + capture), **Projects** (status'd project list), **Gmail** ("needs you" list + reader), **Finance** (compact awareness page — week-to-date daily-spend list + leader-line category donut + bills/flags, confirmed 2026-06-24). The remaining tails (People · Travel · Memory · Knowledge · Health) ship via a descriptor-driven `GenericDomainDetail`. All reads go through CLIENT-core's Rust gateway. Decision 3: spec the end-state; build wires the domains whose backend exists.
→ why: see **docs/findings/cluster-decisions/DECISIONS-LOG.md** (the locked detail-view designs — authoritative, supersedes ADR-028's illustrative seed) + **docs/findings/cluster-decisions/ui.md** §6 (Calendar) §7 (Tasks) + the confirmed mockups `docs/research/mockups/{calendar-page,tasks-page,projects-page}.html` · docs/technical/architecture/app-flow.md (lock rules) · docs/drafts/tauri-client-spec-rewrite.md (decision 3).

## Assumptions
- **CLIENT-card (parallel spec) owns the registry API + the prop seam** this fills: `registerDomain(id, { glance?, detail? })` (CLIENT-card's exported registrar) and `interface DomainDetailProps { domainId: DomainId; onClose: () => void }`. CLIENT-screens calls `registerDomain(id, { detail })` for each domain — it does NOT define a plain `Record<...>`. → impact: Stop (if CLIENT-card's `registerDomain`/`DomainDetailProps` shape differs, reconcile — that is the seam).
- **The canonical domain identity is CLIENT-core's `client/src/domains.ts`** — `DomainId`, `domainLabel(id)`, `domainCluster(id)` are imported from there (CLIENT-core owns the 10-id list, reconciled with CLIENT-theme `relationships.ts` + CLIENT-world clusters). CLIENT-screens does NOT redefine the id/label list; it adds only screens-local **route + lock-tier** meta (`domainRoutes.ts`). → impact: Stop (one canonical id source; screens imports it).
- **CLIENT-core gateway + DTOs are the only data path** (ADR-030): all reads/writes go through `gateway.*` typed `invoke`; the webview holds no token and makes no `fetch`. Review uses `app_review_pending`/`app_review_auto_enabled`/`app_review_approve`/`app_review_reject` (→ `ReviewItem`); Status uses the connection store + `app_lock`/`app_logout`. → impact: Stop.
- **Per-domain read endpoints are the CLIENT-b amendment (brain-side track).** Calendar→`app_calendar_read`, Tasks→`app_tasks_read`, Projects→`app_projects_read` ship now (backends exist: CAL-a/b/c, M8-d-a + M8-d-a2 projects); **Gmail→`app_gmail_read` + Finance→`app_finance_read` are documented but fake-gated** (M8-b1 read-connector blocked on the docling fork; the Finance backend FIN-* is unbuilt); the 5 generic tails' `/app/<domain>` reads are documented DTO contracts the amendment implements when each backend lands. → impact: Stop.
- **The bespoke domain views implement the LOCKED designs verbatim** (DECISIONS-LOG "Expanded-card detail views" + ui.md §6/§7): Calendar = **Month+Week toggle + selected-day panel** (day → its events AND tasks-due; held-tentatives distinct; RSVP strip; **self-only acts inline**, external-effect [invite/RSVP/cancel-w-attendees] **stages a PendingAction → "staged for your review →"** with approve/reject on the **Review** card per ui.md §6=B; morning find-time); Tasks = **Due/Overdue/Upcoming + capture-suggestion confirm tray** (check-off · reschedule · **time-block into a morning focus slot** · accept-suggestion; free-form create deferred to the Ask pop-up per ui.md §7=B); Projects = **status'd project list** (Active/Blocked/Done + target dates + linked tasks); Gmail = **"needs you" urgency list** (sender·subject·why-flagged) + browsable signal mail + search → spotlighted reader + open-in-Gmail (no send v1) + inline accept-task-suggestion / approve-held-event; **Finance = compact awareness page (confirmed 2026-06-24 via `docs/research/mockups/finance-page.html`)** — a **week-to-date daily-spend list** (weekday+date · S$amount; today marked with an **accent divider below it**; upcoming days "—"), a **leader-line category donut** (slices → name/% labels; total in centre), a **transactions** list, a **Bills** strip (inline mark-paid), an **unusual-spend** flag (recategorise/looks-right), a **duplicate-merge** suggestion (keep-both/merge), a **confirm-type** prompt (transfer/income); **all instant local edits, NO GATE** (awareness-only — Artemis never moves money), engine tag `local`. → impact: Stop (these are owner-confirmed; do not redesign).
- Lock rules (app-flow): owner-private data → **Unlocked** (review + all domains); **Status** works in any **Connected** state (it is the lock UI). → impact: Stop.

Simplicity check: bespoke files only for the domains with a LOCKED design (Review/Status/Calendar/Tasks/Projects/Gmail) — distinct content + actions justify them. The 5 remaining tails (People/Travel/Memory/Knowledge/Health) share a descriptor-driven `GenericDomainDetail` (no near-empty bespoke files); each graduates when its design + backend land. Considered redefining the id list locally — rejected; `domains.ts` (CLIENT-core) is canonical (now 11 ids incl. `projects`).

## Specialist Context
### Accessibility
Every a11y requirement is a concrete done-criterion + Task-6 RTL assertion (above), not prose: semantic lists (`ul`/`li`), `<h2>` titles referenced by the overlay's `aria-labelledby`, a keyboard-scrollable (`tabindex=0`) labelled scroll region with `scroll-padding-top`, `role=status`/`aria-live` for loading/empty/error + the approve/reject outcome + the "not yet available" affordance, distinguishing `aria-label`s on per-item Approve/Reject, engine-tag visible text (not colour-only), ≥24×24px Status action buttons, and **focus moved into the re-unlock prompt and returned to the opener**. The CLIENT-card overlay owns the overlay focus-trap + Esc; screens render within it. [FLAG apex-accessibility + apex-ui-ux-design at wave review: confirm the focus move/return + the manual NVDA pass.]
### Security
No token in the webview (ADR-030); all reads/writes via the Rust gateway. Optimistic approve/reject reconciles on the server response (M7-b is the authority; revert on error). Engine tags (`local`/`codex`/`review`) are load-bearing privacy/cost signals — never dropped (design-brief forbidden-pattern). Owner-private domains gated to Unlocked.

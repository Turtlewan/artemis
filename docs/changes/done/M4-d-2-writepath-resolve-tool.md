# Spec: M4-d-2 — Write-path entity wiring (auto-link fact SUBJECTS to PERSON entities, threading `subject_entity_id` through the M4-b A.U.D.N. apply) + the `resolve_entity` cross-module read tool (a memory `ModuleManifest` registered into the `ToolRegistry` at composition)

**Identity:** Wires the M4-d-1 entity backbone into the M4-b write path (every extracted fact's subject resolves to a stable PERSON entity whose id is carried on the fact row) and ships the ADR-013 `memory.resolve_entity` read-tool — a `ToolSpec` inside a memory `ModuleManifest` registered into the M1-a `ToolRegistry` at `compose_brain`, resolving a `{module:"memory", entity_id}` reference to its current facts (logical, ToolRegistry-mediated, never a cross-store join). Entity data layer = M4-d-1; bitemporal apply = M4-a; A.U.D.N. write loop = M4-b.
→ why: see docs/technical/adr/ADR-013-cross-module-links.md (Decision 1 canonical person pointer; Decision 2 logical `{module,entity_id}` ref resolved via the ToolRegistry, NEVER a cross-store join; Decision 6 M4 = Person/Place/Goal backbone) · docs/technical/adr/ADR-004-memory-engine.md (owner-private memory; degrade-don't-crash).

## Assumptions
- **M4-d-1 complete** (the entity data layer): `memory/entities.py` exports `EntityType(StrEnum: PERSON="person"/PLACE="place"/GOAL="goal")`, `EntityRow` (fields: `entity_id: str`, `entity_type: EntityType`, `canonical_name: str`, `external_ref: str | None`, `attributes: str | None`, `created_at: str`, `updated_at: str`), `EntityRef{module: str, entity_id: str}`, `person_fact_key(*, external_ref: str | None, name: str) -> str`, and `class EntityRepository(conn, person_id)` with `resolve_or_create_entity(name: str, entity_type: EntityType, *, external_ref: str | None = None) -> str` (returns a stable `entity_id`), `resolve_alias(text: str) -> str | None`, `add_alias(alias: str, entity_id: str, *, source: str = "extracted") -> None`, `list_aliases(entity_id: str) -> list[str]` (the reverse lookup the resolve tool uses for its `aliases` list), `get_entity(entity_id: str) -> EntityRow` (raises `KeyError` on a missing id), `list_entities(entity_type: EntityType | None = None) -> list[EntityRow]`, `merge_entities(*, keep: str, merge: str) -> None`. The `facts.subject_entity_id` column + the `entities`/`entity_aliases` tables ALREADY EXIST in the M4-a schema as amended by M4-d-1. → impact: Stop (this spec consumes those exact symbols; it does NOT create the entity tables, the `subject_entity_id` column, or `EntityRepository` — those are M4-d-1).
- **M4-b complete** (the write path): `memory/write_path.py` `class MemoryWritePath` constructed with `(repo: BitemporalRepository, embedder: EmbeddingModel, extractor: FactExtractor, decider: AudnDecider, *, candidate_k: int = 5, extractor_model_id: str = "Qwen3.6-27B")` with `async def process_turn(self, text, *, turn_id, role=None) -> WritePathResult` running the per-fact extract→match→decide→apply loop (the apply maps ADD→`repo.add(...)`, UPDATE→`repo.update(fact_key, ...)`, DELETE→`repo.tombstone(fact_key)`, NOOP→nothing); `ExtractedFact { subject, relation, object, confidence, keywords, contextual_description }`; `AudnDecision { op, target_fact_id, object, confidence }`; the `build_write_path(store, model)` factory in `memory/__init__.py`. M4-b's `process_turn` per-fact loop iterates `for ef in facts:` (each `ef` an `ExtractedFact`), and the apply step maps ADD→`repo.add(...)`, UPDATE→`repo.update(fact_key, ...)`, DELETE→`repo.tombstone(fact_key)` (verified against M4-b Task 3); each fact's apply is already wrapped in a per-fact try/except (degrade-don't-crash: a failing fact increments `errors` and continues, never aborting the batch). → impact: Stop (this spec MODIFIES that loop to resolve `ef.subject`→PERSON entity BEFORE the apply and thread `subject_entity_id` into the `repo.add`/`repo.update` calls; it reuses the existing per-fact try/except shape).
- **M4-a repository** `BitemporalRepository.add(...)` / `update(...)` and `FactRow` are the M4-a symbols; `add`/`update` are extended HERE with a `subject_entity_id: str | None = None` kwarg written to the M4-d-1 `facts.subject_entity_id` column, and `FactRow` gains a `subject_entity_id: str | None` field. Backward-compatible: default `None` ⇒ existing M4-a/M4-b tests are unaffected (no caller is forced to pass it). → impact: Stop (the column already exists from M4-d-1; this spec only adds the kwarg/field that writes+reads it, plus the `facts_for_entity` helper).
- **M1-a tool/registry contract:** `ModuleManifest`, `ToolSpec(name, description, args_schema: type[BaseModel], return_schema: type[BaseModel], callable_ref: Callable[..., BaseModel], action_risk: ActionRisk)`, `DataScope.OWNER_PRIVATE`, `ActionRisk.READ`, `ToolRegistry` is constructed with an `EmbeddingModel` (M0-d port) + an optional `VectorStore` defaulting to `InMemoryToolIndex` (M1-a Task 3 — i.e. `ToolRegistry(embedder)` is valid; the test's `ToolRegistry(FakeEmbedder())` relies on this default), with methods `register(manifest)` / `get_tool(fq_name) -> ToolSpec` / `retrieve_tools(query, k)` (from `artemis.manifest` + `artemis.registry`). The `resolve_entity` callable takes a validated args model and returns a return model (the `ToolSpec.callable_ref` contract); both args/return schemas are Pydantic `BaseModel` subclasses. → impact: Stop (the memory manifest + tool MUST satisfy this exact contract; `get_tool("memory.resolve_entity").callable_ref` is the dispatch route ADR-013 Decision 2 mandates).
- **ADR-013 Decision 2 — logical reference, never a cross-store join:** a cross-module reference is a `{module, entity_id}` tuple resolved by calling the TARGET module's tool through the `ToolRegistry`. `resolve_entity` ONLY resolves memory-homed entities — it asserts `args.module == "memory"` and refuses any other module (a non-memory ref is resolved by THAT module's own tool; `resolve_entity` never reaches across scopes/stores). → impact: Stop (the `module=="memory"` guard is load-bearing — it is the structural enforcement of "no cross-store reach"; resolution stays within the owner's keyed memory DB).
- **Subject-resolution scope = PERSON entities ONLY (deliberate boundary, not a gap):** M4-d-2 auto-links fact SUBJECTS to PERSON entities (the ADR-013 Decision 6 headline person pointer / Decision 1 canonical person pointer). PLACE and GOAL entities are fully supported by the M4-d-1 data layer (`EntityType.PLACE`/`GOAL`, `EntityRepository`) AND by the `resolve_entity` tool (it resolves any entity_id regardless of type), but they are created ON-DEMAND by their owning spokes (Productivity→Goal, Travel/Maps→Place per ADR-013 Decision 6 "detailed schema deferred to each implementing spec"), NOT auto-extracted on the memory write path here. The write path only ever calls `resolve_or_create_entity(ef.subject, EntityType.PERSON)`. → impact: Stop (state this boundary explicitly in code comments + this spec; do NOT add PLACE/GOAL extraction to the write path).
- **The canonical-self ("owner") entity:** every fact subject resolves to a stable PERSON entity; the subject `"owner"` (the M4-b extraction default for first-person statements) resolves to THE canonical-self PERSON entity whose `canonical_name == "owner"`. M4-d-1's `resolve_or_create_entity("owner", EntityType.PERSON)` returns the same stable id on every call (create-once, resolve-thereafter) — so repeated owner facts all carry the SAME `subject_entity_id`. → impact: Stop (the test asserts the lives_in fact's `subject_entity_id` resolves to a PERSON entity with `canonical_name == "owner"`, and that an UPDATE to the same subject keeps the same entity id).
- **Degrade-don't-crash on entity resolution** (inherits the M4-b contract): the subject→entity resolution is wrapped so that an entity-resolution failure does NOT break the turn — on failure, fall back to `subject_entity_id = None`, log (no fact plaintext), and continue applying the fact (the fact is still stored, just without the entity link). This reuses M4-b's per-fact try/except spirit but specifically guards the resolution step so a resolution error degrades the link, not the whole fact. → impact: Stop (a fact must never be lost because entity resolution failed; the link is best-effort).
- **The resolve_entity callable needs an owner memory store handle** — it is bound via a thin factory closure over an injected `SqliteMemoryStore` (mirroring how other stateful module callables will be bound). `memory_manifest(store: SqliteMemoryStore) -> ModuleManifest` closes `resolve_entity` over the store (the store lazily opens the keyed DB and exposes the repo + an `EntityRepository`); the bound callable reads from THAT owner store only. → impact: Caution (the store handle injection is the documented binding pattern; keep it a thin closure — no global singleton, no second store).
- **Composition seam = `compose_brain` in `src/artemis/gateway.py`** (VERIFIED: M1-c Task 1 creates `compose_brain` in `gateway.py`; M1-d registers module manifests there via `registry.register(...)`; M4-c extends `compose_brain` to conditionally build the owner `SqliteMemoryStore` + `MemoryWriteQueue` when an unlocked owner keyed DB is available). That one function holds both the `ToolRegistry` and the owner store — so it is the single registrar: when memory is active, call `registry.register(memory_manifest(store))`. → impact: Stop (do NOT invent a new composition file; the registrar is `compose_brain` in `gateway.py`. The one build-time check: register at the exact point in the memory-active branch where M4-c bound the constructed `SqliteMemoryStore` and the `registry` is in scope — if M4-c factored the owner-store build into a helper, register inside that helper (same `gateway.py`), passing the registry in. Same file either way.)
- Off-hardware: deterministic. Reuse the M4-b fixture (the M4-a DB fixture — real keyed SQLCipher or plain-sqlite+sqlite-vec fallback — + `FakeEmbedder` + `FakeExtractor`/`FakeDecider`). Entity resolution + the `resolve_entity` tool are model-free (pure DB lookups). → impact: Stop (no model call; no on-hardware gate).

Simplicity check: considered a separate post-write entity-linking pass over the facts — rejected: the subject is already in hand inside the per-fact loop, so resolving it inline before the apply is one indexed lookup with no second iteration (and keeps the link transactional with the fact write). Considered auto-extracting PLACE/GOAL entities here too — rejected: ADR-013 Decision 6 homes their creation in their owning spokes; the memory write path only owns the PERSON person-pointer (Decision 1). Considered a global memory store singleton for the tool callable — rejected: a thin closure over the injected `SqliteMemoryStore` at `compose_brain` is the M1 binding pattern (no global live-store singleton). Considered resolving non-memory refs inside `resolve_entity` — rejected: ADR-013 Decision 2 forbids cross-store reach; each module resolves its own refs, so the `module=="memory"` guard is mandatory. This is the minimum that satisfies ADR-013 Decisions 1+2+6 on the M4-a/b/d-1 substrate.

## Specialist Context
### Security
`resolve_entity` is **READ-only** (`action_risk = ActionRisk.READ`) and **owner-private** (the manifest `data_scope = DataScope.OWNER_PRIVATE`). **The real access gate is NOT the manifest label** — per M1-a, `data_scope`/`action_risk` are *advisory* on the registry in M1 (dispatch-time scope enforcement is the deferred M2/M5 `[FLAG apex-security]` in M1-a). The effective control here is **registration-gating + the keyed-store binding**: `memory_manifest(store)` is only registered when M4-c has built an unlocked owner `SqliteMemoryStore` (a locked/guest scope has no store → the tool is never registered → not callable), and the bound store IS the owner's keyed SQLCipher DB, so resolution physically cannot cross the M2 wall regardless of the label. The `module == "memory"` guard is two-layered: the `Literal["memory"]` rejects a wrong module at pydantic validation, and a runtime assert backs it (ADR-013 Decision 2 — no cross-store reach; a non-memory ref resolves through THAT module's own tool, never a join). The tool returns owner facts — **never log fact plaintext or entity names** at info level (ids/provenance only). A forged/guessed `entity_id` cannot probe across scopes (the store is the owner's keyed DB) and an unknown id raises `EntityNotFound("entity not found")` — a typed error whose message omits the queried id (no probe-confirmation side-channel). Write-path entity resolution inherits M4-b's degrade-don't-crash (failure → `subject_entity_id = None`, fact never lost, no raise). [FLAG for apex-security (M4 gate): confirm registration-gating + keyed-store binding are the effective wall (manifest label is advisory in M1); confirm the two-layer `module` guard; confirm no fact plaintext / entity name is logged or embedded in `EntityNotFound`; confirm dispatch-time `data_scope` enforcement is tracked as the M1-a deferred item.]

### Performance
Subject resolution is ONE indexed alias/point lookup per extracted fact (`resolve_or_create_entity` over the M4-d-1 `entities`/`entity_aliases` indexes) — and it runs on the ASYNC write-path worker (M4-b), OFF the interactive turn, so it adds no response latency. `facts_for_entity` is an indexed scan on `facts.subject_entity_id` (M4-d-1 indexed the column). `resolve_entity` is a bounded read (one entity row + its aliases + its current facts) — ADR-013 flags this read-path as HOT (called on most cross-module operations); at per-person scale (thousands–tens-of-thousands of facts, ADR-004) the indexed lookups are well within budget.

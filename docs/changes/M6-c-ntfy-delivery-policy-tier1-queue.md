# Spec: M6-c — ntfy delivery adapter (priority/tags/quiet-hours-delay/action-buttons over the M0-b ntfy service) + owner-tunable threshold/policy config + dedup store + the durable Tier-1 queue (deferred sensitive hooks → run on next owner unlock)

**Identity:** Implements the `deliver(list[OutboundMessage])` sink M6-b calls: a `NtfyDelivery` adapter that maps each `OutboundMessage` to an ntfy publish (priority, tags, click URL, action buttons) against the M0-b local ntfy LaunchDaemon, applies owner-tunable policy (per-area noisiness thresholds, quiet hours → delay/hold `deferrable`+`digest` messages, what surfaces), a persistent dedup store (suppress an already-sent `dedup_key`/`dedup_value`), and the durable Tier-1 queue that receives M6-a's skipped sensitive hits and drains-and-runs them on the next owner-unlocked session.
→ why: see docs/technical/architecture/brain.md § "Proactive engine (Heartbeat)" (Delivery via ntfy: priority/tags/quiet-hours-delay/action-buttons; owner-tunable thresholds) · docs/technical/adr/ADR-006-two-tier-proactivity.md (Tier-1 hooks queued for the next owner-unlocked session) · docs/technical/adr/ADR-005-owner-key-broker.md (owner-unlock session lifecycle).

## Assumptions
- M6-b complete: `src/artemis/proactive/hit_handler.py` exports `OutboundMessage` (`title`, `body`, `urgency`, `disposition: Literal["immediate","deferrable","digest"]`, `tier`, `delivery: DeliverySpec | None`, `dedup_key`, `dedup_value`, `source`) and the `HitHandler` calls an injected `deliver: Callable[[list[OutboundMessage]], int]` (returns the count of successfully published messages — B2 fix; `int`, not `None`). → impact: Stop (M6-c IS that `deliver` sink; `NtfyDelivery.__call__` returns `int`; signature must match M6-b exactly).
- M6-a complete: `src/artemis/proactive/hook_types.py` exports `Hit`, `DeliverySpec`; the `Heartbeat` exposes a `tier1_sink: Callable[[Hit], None]` seam and re-runs hooks when called. → impact: Stop (M6-c's queue IS the `tier1_sink`; the drainer re-invokes the hooks' `check_ref`).
- M0-b complete: the **ntfy LaunchDaemon** runs locally on `127.0.0.1:{NTFY_PORT}` (`{NTFY_BIN} serve --listen-http 127.0.0.1:{NTFY_PORT}`); the ntfy port is in `Settings`/`paths`. → impact: Stop. RESOLVED (gate 2026-06-08): M6-c reads the ntfy base URL from `Settings` via `ntfy_base_url(settings) -> str` (default `http://127.0.0.1:{ntfy_port}`); if `Settings` lacks `ntfy_port`, M6-c adds a typed one (M0-a default). **Topic = `f"artemis-{slot}-{topic_secret}"`, where `topic_secret` is a cryptographically-random hex (≥16 bytes) generated at slot-init and stored in the slot's non-public Settings — the topic IS the egress capability and MUST NOT be guessable (BLOCK fix).** [on-hardware confirm only: whether M0-a/M0-b already provide `ntfy_port` + the secret slot, else M6-c introduces them.]
- ntfy publish is an HTTP POST to `{base}/{topic}` with headers `Title`, `Priority` (min|low|default|high|max), `Tags` (comma list), `Click` (URL), `Actions` (the ntfy action header syntax) and the body as the POST body — the ntfy server's standard publish API. → impact: Caution. RESOLVED (gate 2026-06-08): draft the ntfy `Actions` header per ntfy docs (`view`/`http` action form) behind `_render_actions(...)` (the single point to adjust); off-hardware tests assert the rendered header shape against the documented format, and the exact serialization is verified against the installed ntfy version at the GATED on-hardware Task 7.
- M2 unlock signal: a `KeyProvider.is_owner_unlocked() -> bool` + a way to be NOTIFIED on unlock. M6-c needs to drain the Tier-1 queue WHEN an owner session unlocks. → impact: Caution. RESOLVED (gate 2026-06-08): drain-on-tick-when-unlocked is the M6 contract — poll `is_owner_unlocked()` at the start of each tick; no new M2 surface. Drain latency ≤ one granularity tick after unlock (acceptable). An optional M2 `on_unlock` event callback is a later enhancement, out of M6 scope.
- The dedup store + the Tier-1 queue are **small local persistent stores**. They hold Tier-0-safe metadata only (dedup keys/values; queued *hook identities* — module.hook_name + when-queued — NOT sensitive payloads, since a Tier-1 hook's `check_ref` is NOT run while locked, so there is no sensitive data to store; it runs only on drain-after-unlock). → impact: Caution. RESOLVED (gate 2026-06-08): dependency-free JSON-file stores under `<slot>/proactive/` (`dedup.json`, `tier1_queue.json`, `tier1_dead.json`, `held.json`), **same-dir atomic temp+`os.replace`** (temp created in `target.parent` — a cross-device temp breaks `os.replace` atomicity on APFS), **corrupt-load → warn + start empty** (never crash the scheduler). They hold ONLY Tier-0-safe metadata (no sensitive payload ever lands here — Tier-1 checks don't run while locked), so no encryption (matches ADR-006). Swappable behind the `DedupStore`/`Tier1Queue` interfaces.
- Owner-tunable policy: a typed config (quiet-hours window, per-area/per-module noisiness thresholds, global mute, "what surfaces" allow/deny by urgency) loaded from a config file under the data dir, with documented defaults — owner-editable. → impact: Low (M6-c ships defaults + the loader; the owner edits the file; no UI in M6).

Simplicity check: considered using the ntfy Python client library — rejected; ntfy publish is a single HTTP POST with headers, so a thin `httpx`/`urllib` POST is the minimum and avoids a dependency (the project already has an HTTP client from M1-b's OpenAI adapter). Considered encrypting the dedup/queue stores — rejected; they hold only Tier-0-safe metadata (no sensitive payloads ever land there because Tier-1 checks don't run while locked), so a plain atomic JSON file is sufficient and matches ADR-006's "minimised, mostly-derived" framing. Considered an unlock event subscription into M2 — rejected for M6 scope; drain-on-tick-when-unlocked needs no new M2 surface.

## Specialist Context
### Security
M6-c is the proactive engine's EGRESS surface (the only place a notification leaves the box). Invariants the build MUST honour: (1) the dedup + Tier-1 queue stores hold ONLY Tier-0-safe metadata (dedup keys/values + queued hook IDENTITIES) — NEVER a Tier-1 hook's payload, because a Tier-1 `check_ref` is not run while locked (it runs only on drain-after-unlock, and its resulting message follows the normal delivery path then); (2) a Tier-0 notification body must not carry raw sensitive data — Tier-0 hits derive only from the minimised corpus by construction (M6-a's `OWNER_PRIVATE ⇒ tier==1` guard), so the body is already Tier-0-safe, but the build should NOT echo arbitrary payload fields that weren't intended for an over-the-wire notification; (3) ntfy binds `127.0.0.1` (M0-b) — delivery to the owner's device rides the existing ntfy/Tailscale path, not a public listener; (4) the policy `muted`/quiet-hours/floors are owner controls, not security controls — they reduce noise, they do not gate sensitive data (that is the tier wall in M6-a). [HARD FLAG for the apex-security gate: (a) ADR-006 proactive-key/Tier-0-corpus scope-creep — audit that Tier-0 notification bodies cannot leak more than the minimised corpus intends; (b) the Tier-1 queue must persist NO sensitive payload — verify the serialised JSON contains only `{module, hook_name, queued_at}`; (c) the ntfy topic name is a capability (anyone who knows it can publish/subscribe) — confirm the topic is not guessable / rides access-controlled ntfy + Tailscale.]

### Performance
Quiet-hours-hold + batch-low→digest (M6-b) + dedup minimise notification volume and device interruptions. The dedup/queue JSON stores are tiny (Tier-0 metadata) — atomic-write cost is negligible per tick. drain-on-tick-when-unlocked adds one `is_owner_unlocked()` poll + (only when unlocked + non-empty queue) the queued `check_ref` runs — no cost while locked or when the queue is empty.

### Accessibility
(none — delivery is to ntfy clients whose own a11y is out of scope; notification copy is M6-b's concern)

# Spec: CLIENT-f — ArtemisMac app target (XcodeGen macOS target + MenuBarExtra/NSPanel/Window/Settings scene graph + macOS AppModel coordinator + reused CLIENT-e screens + Mac pairing)

**Identity:** Implements the separate native macOS app target `ArtemisMac` on top of ArtemisKit: a XcodeGen macOS target (Developer-ID, Hardened Runtime, KeyboardShortcuts dep) and the Athena-style scene graph — `MenuBarExtra(.window)` home + a global-hotkey floating `NSPanel` + a singleton `Window(.hiddenTitleBar)` for full history + a `Settings` scene — driven by a macOS `AppModel` coordinator over ArtemisKit's `Authenticator`, reusing the CLIENT-e Review/Chat/Status screens.
→ why: see docs/technical/adr/ADR-017-macos-client-surface.md (target structure, Athena scene, auth path, Developer-ID distribution).

## Assumptions
- CLIENT-c (ArtemisKit) complete AND amended with the macOS auth code path: `Authenticator` (`pair`/`connect`/`unlock`/`connectAndUnlock`/`logout`/`lock`/`currentToken`/`isPaired`), `ApiClient`, `DeviceIdentity`/`Signer`, `KeychainStore` — all platform-agnostic and compiling for macOS. The Mac auth deltas (data-protection keychain, passcode/`deviceOwnerAuthentication` fallback, modality-aware prompts) land in ArtemisKit via a SEPARATE CLIENT-c amendment, NOT in this spec — this spec only DEPENDS on the existing `Authenticator`/`Signer` seam. → impact: Stop (the Mac app imports + drives these exactly; it adds no ArtemisKit code).
- CLIENT-e (screens) complete: `ReviewScreen`/`ChatScreen`/`StatusScreen` + their `@Observable` view-models + the app-local `BrainApi` protocol (`extension ApiClient: BrainApi`), constructed from `model.api` + a token provider + `onLocked`. These screen files live in the iOS app's `Sources/Screens/`; this spec **reuses them by adding the CLIENT-e `Sources/Screens` folder to the ArtemisMac target's source list** (shared, not copied). → impact: Stop (the Mac scenes embed these exact screen views).
- **Swift 6 language mode ON**; view-models/coordinators are `@MainActor @Observable final class`; no `ObservableObject` (apex-swift hard blocks). → impact: Stop.
- **Separate native macOS target, AppKit-backed SwiftUI** (ADR-017 §Decision 2): NOT a single multiplatform target, NOT Mac Catalyst, NOT "Designed for iPad on Mac". All platform divergence lives in the view/scene layer (`Sources/macOS` + leaf `#if os(macOS)`); ArtemisKit stays untouched. → impact: Stop.
- **The floating summon surface MUST be an `NSPanel`** (`NSStatusItem` + `NSHostingView`, hotkey via the `KeyboardShortcuts` SPM dep) — SwiftUI `WindowGroup`+`openWindow` misbehaves for float-on-top across Spaces/full-screen (ADR-017 §Decision 3 / research Q2). → impact: Stop (the global-hotkey scene is AppKit-bridged, not a SwiftUI scene).
- **The Mac is just another paired device** (ADR-017 §Decision 4, ADR-010 unchanged): it generates its own SE P-256 keypair at pairing via ArtemisKit's `DeviceIdentity`; registry already supports N devices — additive, no protocol change. → impact: Caution (Mac pairing reuses CLIENT-c's `pair`→`connectAndUnlock` path verbatim).
- **Distribution: Developer ID + notarization + Hardened Runtime**, entitlements = Keychain Sharing + network-client; App Sandbox skipped (ADR-017 §Decision 6). Signing/notarization run only on a real Mac → GATED. → impact: Caution (the XcodeGen settings declare these; notarization is a gated manual step recorded in handoff).
- Pairing input on Mac: **manual code entry is the always-available baseline** (no `DataScannerViewController` on macOS — that's iOS-only; the QR convenience path does not port). The code is shown out-of-band by the brain (CLIENT-b `mint_pairing_code`). → impact: Caution (Mac onboarding is manual-entry only).
- apex-swift Verification Recipe is v0.1.0 unvalidated → the first Mac build validates it (→ 1.0.0). → impact: Caution (recorded).

Simplicity check: considered a single multiplatform target with a Mac destination — rejected per ADR-017 (Developer-ID/outside-App-Store + divergent scene graph → separate target is Apple's own guidance, and a shared scene body would be `#if`-soup). Considered a SwiftUI `WindowGroup`+`openWindow` floating surface — rejected (research Q2: misbehaves on Spaces/full-screen; `NSPanel` is the idiomatic path). Considered copying the CLIENT-e screen files into the Mac target — rejected; the screens are size-class-adaptive and reused by adding the same `Sources/Screens` folder to both targets' source lists (one source of truth). Considered a fresh Mac AppModel from scratch — rejected; reuse the CLIENT-d state-machine pattern (same protocol-existential `Authenticating` seam, same transitions).

## Parked / hardware-gated
Two ADR-017 auth unknowns are NOT planning blockers — they are build-time verification items resolved on the first real Mac build (Task 7) and recorded in handoff:
1. **Mini biometric-less fallback** — a Mac with no Touch ID (the Mini, or any desktop without a Touch-ID Magic Keyboard) must authenticate via `deviceOwnerAuthentication` / login-password fallback rather than biometrics. The fallback itself lives in the CLIENT-c macOS auth path; THIS spec verifies on hardware that `connectAndUnlock` succeeds on a biometric-less Mac via passcode. **Security posture (accepted downgrade):** passcode-only is a known authenticator downgrade (NIST AAL1 vs the biometric+enclave AAL2 path) — **accepted** for the single-owner personal appliance; Settings should surface a one-line notice when the device has no biometric so the owner knows biometric gating is not active. (ADR-017 §Decision 5b / §Consequences.)
2. **macOS 26 `.or` passcode prompt behaviour** — confirm `[.biometryCurrentSet, .or, .devicePasscode]` prompts as a true alternative (biometric OR passcode), not the documented-quirk sequential (biometric THEN passcode), under macOS 26 on real hardware. (ADR-017 §Decision 5d / research Q3.)

## Specialist Context
### Security
The Mac is just another paired device (ADR-010 unchanged): its own SE P-256 keypair, registered with broker + brain registry; the SE key is non-exportable and biometric/passcode-gated by ArtemisKit. The app never persists a secret outside the keychain (ArtemisKit owns that); `MacAppModel` holds only transient UI state + a redacted `lastError`. The pairing code is entered, used once, never stored or logged. Distribution is Developer-ID + notarized + Hardened Runtime; entitlements are limited to Keychain Sharing + network-client; App Sandbox is skipped (personal appliance; sandbox complicates the keychain-access-group the auth layer relies on — ADR-017 §Decision 6). [FLAG apex-security + apex-auth: confirm the entitlements file is minimal (no over-broad grants), the keychain-access-group matches the CLIENT-c data-protection-keychain group exactly, no token/code/nonce is rendered or logged, the floating panel does not leak content to screen-capture/recording surfaces, and the biometric-less passcode fallback gates key use as tightly as the biometric path.]

### Performance
`@Observable` re-renders only the views reading changed properties. The `NSPanel` hosts a single `NSHostingView`; the global hotkey is an event-tap via `KeyboardShortcuts` (no polling). Connect+unlock is two IPC-backed round-trips behind one Touch-ID/passcode gesture. The `Authenticator` is an actor — no blocking work on `@MainActor`.

### Accessibility
apex-accessibility applies (Mac is a first-class UI surface). Mac-specific requirements: full keyboard operability (every action reachable without the mouse — menu commands + ⌘-shortcuts; the global summon hotkey is itself a keyboard affordance and is user-rebindable); VoiceOver labels on all MenuBar/panel/window/Settings controls; the floating panel takes and returns focus correctly on summon/dismiss; lock state carries text+icon, never colour alone; Dynamic Type / named text styles throughout; Reduce Motion honoured for panel/summon transitions. The reused CLIENT-e screens carry their per-screen a11y criteria unchanged.

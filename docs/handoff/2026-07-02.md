## Specialist Flags

### 🐞 BUG for bug-fixing mode (owner-flagged, do NOT fix now)
**R3 web_q synth over-applies report-and-attribute → hedges instead of answering.** For general web
questions with an obvious answer, the synth lists every minor cross-extract disagreement and gives
NO clear answer. Owner examples:
- "latest stable python version" → reported 3.14.3 vs 3.14 vs 3.14.0 vs 3.13.5 across sources, ended with a hedge instead of "3.14.x".
- "what is the latest claude version" → listed Sonnet 5 / Opus 4.8 / Opus 4.7 / Fable 5 with no clear answer.
Root: the report-and-attribute conflict policy (ADR-038, built for the eval's *conflicting* category)
is applied to ALL web_q. Fix (bug-fixing): the synth should give a CLEAR answer when extracts
substantively agree or an authoritative source exists (e.g. python.org), reserving report-both for
GENUINE unresolved conflicts. Synth-prompt tuning in web_tool.py `_SYNTH_SYSTEM`; the webtool-eval
harness/calibration can measure the tuning. Related: classifier-prompt tuning (intent.py).

### Exe UI fix (done this session)
The built exe rendered the OLD UI while `tauri dev` showed the GOOD one — SAME source. Cause: per-
origin localStorage — the exe's `tauri://` origin held stale UI/theme state from the June-30 build;
dev's `localhost:5173` origin was clean. Fix: cleared `%LOCALAPPDATA%\com.artemis.client\EBWebView`
+ relaunched the release exe → good UI. NOTE for future rebuilds: a fresh `tauri build` alone does
NOT clear this per-origin state; clear EBWebView (or bump a state version) when the UI looks stale.

## Blocked Actions (→ planning must resolve)
- **Write a "chrome-capable FetchSandbox" spec (isolate substrate).** Two production-isolate defects,
  both diagnosed + fix-confirmed end-to-end (real render, exit 0) — full detail in
  `docs/progress/js-rendering-fetcher.md` (gitignored) + memory `fetchsandbox-caps-fatal-to-chrome`:
  1. **`run_isolated` arg-quoting** — command tokens reach WSL's `bash -c` unquoted; a URL with `()`
     (or a path with a space) is a shell syntax error. Fix = `shlex.quote()` the positional args
     (also a shell-injection hardening). Low risk, pure correctness.
  2. **`FetchSandbox.run` render caps profile** — it hardcodes `SandboxCaps()` (512MB / 1 CPU / 128
     pids / 512MB VSZ); chrome needs unlimited VSZ + 1.5GB RAM + 4 CPU + 256 pids or it SIGTRAPs. Add
     an opt-in caps profile (default stays 512MB; `JsFetcher` opts in). **Needs an apex-security review
     of the enlarged resource envelope** — containment (egress/de-priv/mount-pid isolation) is
     unchanged; only the DoS ceiling grows, bounded by cgroup `memory.max`. This is why the owner chose
     review-first (option B).

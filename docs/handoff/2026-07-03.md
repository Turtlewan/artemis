## Decisions Made
- **Arg-quoting fork resolved as Option A (harden in place), NOT Option B (base64 rewrite) — because the owner was away.** Two dual-pass apex-security reviews BLOCKed Codex's initial guest-side arg-decode as fail-OPEN. Option B (base64 side-channel) is a BIG fork (changes the spec's whole approach + forces modifying a pre-existing test), so per the fork protocol I did NOT do it unilaterally while the owner was away — I applied Option A (fail-closed hardening of the existing decode). Re-review dual-pass → both CLEAN. **Owner later chose "commit + spec Option B"** → Option B is now specced (`argv-base64-side-channel`, ADR-042) but NOT built; owner to greenlit. ⚠️ review needed: decide whether to build Option B.
- **The spec's literal fix was incomplete — kept + hardened Codex's out-of-spec decode block.** `chrome-capable-fetch-sandbox` Assumption 2 claimed `shlex.quote()` alone suffices (WSL's `bash -c` reparse "consumes" the quoting). Host probes (`/tmp/wsl_probe*.py`) disproved that: the quotes arrive at the guest LITERALLY and must be stripped guest-side. Codex had correctly added a guest decode; I verified it's load-bearing (didn't delete it) and hardened it. ADR-041/042 record the true mechanism.
- **Stub-sync scope deviation (mechanical, no review).** The `FetchSandbox.run` `caps_profile` signature change broke three out-of-spec test stubs that subclass it (`test_invoke.py`, `test_ask_routes.py` — committed with the fix; `test_js_fetch.py` — committed with js-rendering-fetcher). Same recurring sync the prior `js-fetch-output-limit` handoff flagged; the spec's Files-to-Change under-lists these stubs every time `FetchSandbox.run`'s signature changes.
- **Tasks 2/3 built inline as Opus, not dispatched to Codex.** Small mechanical resource-cap plumbing; not worth the subprocess overhead. Task 1's security-critical part was also finished inline (the fork + hardening needed the reasoning context).

## Specialist Flags
- **WSL argv passing has TWO interacting quoting layers and is content-dependent** (Windows `list2cmdline` framing + WSL's `bash -c` reconstruction). A no-space value with parens (`…/Python_(programming_language)`) raw → `bash` syntax error; quoted → arrives clean. A value with spaces → arrives WITH literal quotes attached. This is why the decode is load-bearing and why Option B (never put untrusted values on the command line) is the sturdier design. Filed to memory `wsl-argv-quoting-two-layer`.
- **Fail-open in bash is easy to write and hard to spot.** The original decode used `< <(python3 …)` process substitution + a `while read` loop — BOTH exempt from `set -e`, so a crashed/partial decode silently ran a truncated command. The fix: plain pipeline `{ … } > file || abort`, `surrogateescape`, explicit argv-count check. When a bash change sits under a `# fail CLOSED` invariant, scrutinize process-subs and loop conditions specifically.

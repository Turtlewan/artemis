## Decisions Made (the WHY)

1. **Gmail = read-only mirror, split-by-depth ingestion** (`modules/gmail.md`). Metadata-index ALL mail;
   full body + attachments + memory only for signal categories (Primary/Updates/Forums). Gmail's own
   category label (kept as metadata) gives marketing-vs-real differentiation *for free* → no need to
   deep-read junk; minimises injection surface + memory pollution + Mini cost. Bounded 12-mo backfill,
   History-API sync, end-state 3-stage urgency hook. Owner chose the fullest hook.

2. **Productivity = Tasks + Projects + Areas** (`modules/productivity.md`, all forks LOCKED): full 3-level
   Calendar time-blocking (but the gap-fill + completion-check *hooks* opted out — capability stays);
   capture = **suggestion-inbox you guide → written automation recipes** (the owner's framing "I guide
   it at the start but the automation rules get written" maps exactly onto the M7 recipe loop + the
   ADR-012 recurrence→recipe bridge — the rules ARE recipes, inspectable, not a black box); **no
   Google-Tasks / no external integration at all**; **both** recurrence modes; hooks = Morning-plan +
   Overdue + Weekly-review. Habits/Goals deferred (the time-blocking rail is reserved for them).

3. **★ Gated-action staging — the architecture gap, RESOLVED → ADR-012 + GATE-a/b.** Building CAL-b
   revealed the owner-approval surface (M7-b `ReviewSurface` + CLIENT-b `/app/review/*`) is entirely
   **recipe**-based: a recipe is a reusable learned-behaviour *template* (`approve(name)` = "may auto-fire
   in future"). A gated calendar write is a one-off *instance* with bound args (`approve` = "execute this
   once"). Conflating them is a category error. **Decision (ADR-012):** one-off actions are **pending
   actions** in a dedicated `PendingActionStore` + `ActionStagingService` (stage → owner approves on the
   Review screen → execute-once via the ToolRegistry). The two surfaces are complementary — pending
   actions = *permission now*, recipes = *automate later* — and the recurrence of staged actions is the
   signal that feeds the recipe-learning loop. This was the headline blocker for the whole write path +
   every future external-effect spoke; it is now built (GATE-a backend + GATE-b client).

4. **Module-layout convention LOCKED:** domain modules under `src/artemis/modules/<name>/` (calendar,
   gmail, productivity); shared Google auth stays in `integrations/google/` (infra). M8-b1 was migrated
   from `integrations/google/gmail/` → `modules/gmail/` (auth imports preserved).

5. **AFK drafting via parallel background agents (autonomous).** Two waves; agents drafted + self-reviewed
   + ran the gate + parked-on-seam-mismatch, then a reconciliation wave bound every parked seam to the
   real (now-existing) symbol. The "park don't guess" discipline is what surfaced ADR-012 instead of a
   broken guessed binding — worth preserving.

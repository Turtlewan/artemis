## Specialist Flags (gotchas git diff won't surface)
- **Timezone correctness in cron is load-bearing and was a real bug.** First impl fed croniter a *naive* local datetime; `get_next(float)` then returned an epoch desynced by the UTC offset (owner is SGT, +8h) ΓÁÈ jobs computed ~8h off and never became due. Fix: feed croniter a **tz-aware local** datetime (`datetime.fromtimestamp(base).astimezone()`) so cron is local-wall-clock ("0 7 * * *" = 07:00 **local**) and the epoch round-trips. See `scheduler/_cron_next`.
- **Windows near-epoch `astimezone()` quirk:** `datetime.fromtimestamp(1000).astimezone()` raises `OSError [Errno 22]` on Windows for tiny (ΓÊê1970) timestamps. Production never hits this (`base` is always `time.time()`); the fix was to use realistic epochs in tests (`T0 = 1.9e9`), **not** to gold-plate the production path. Flag for any future time-based test on Windows: don't use small synthetic timestamps with `astimezone()`.
- **Catch-up semantics = fire-once-if-overdue, then advance** (not replay-every-missed-occurrence). A job due during downtime fires once on the first post-reboot tick; cron jobs then advance to the next future slot. Documented inline in `_fire`.

# Handoff — 2026-07-01: capability-build slice (build-by-chat) — built + proven live

The dogfood loop now works in the real app: type a build request in the Ask popup → gate the plan → watch the real sandbox verify → gate the result → it's installed. Specs `CB-1, CB-2, CB-3, CB-4, CB-4.1, CB-5a` all in `done/`; details + commits are in `docs/status.md` and `git log`. This handoff is the narrative git/status can't reconstruct.

## Decisions Made
- **The network guard is an AST import-level scan, NOT a security boundary** — locked as the slice's load-bearing contract (`scan_for_unsafe_imports`, written into the CB-1 spec). It enforces the *product* rule "network/secret capabilities wait for WSL2," and keeps honest capabilities honest; a determined model could evade it (`os.system`/`getattr`/`eval`). **WSL2 isolation remains the real boundary** before any network-touching capability runs. Do not let anyone treat the scan as containment.
- **First dogfood capability is network-free by choice** (owner). The engine already proved it can author IMAP-class code (the 2026-06-30 `email_intake` forge test); the chat layer doesn't need a network capability to prove itself, so we kept WSL2 + the secret-paste gate out of this slice.
- **Engine-first sequencing, client-against-built-brain.** Brain (CB-1/CB-2) first, then the gateway (CB-3), then UI (CB-4) — the same pattern as the CR-* revival. CB-3/CB-4 specs were frozen only *after* CB-2's wire shape was built + live-smoked, which avoided churn.
- **Build-stream wire = named SSE events** (`event: status` / `event: result` / `[DONE]`), deliberately distinct from `/app/ask`'s bare-`data:` text stream; CB-3 added a separate `parse_build_frame` rather than overloading the ask parser. In-flight proposals live in an **in-memory** dict on `app.state` (not durable — a build straddling a brain restart is lost; acceptable for a single-user hub).
- **CB-5 split: data (CB-5a, built) vs node visual (CB-5b, deferred).** The map-node visual is blocked on an owner design decision ("what a capability node looks like"). **Owner directive: fan out display-research agents before speccing CB-5b** (parallel research pre-authorized). CB-5a deliberately ships *only* the brain `/app/capabilities` list — the Tauri list command + TS wrapper were left for CB-5b as dead-until-consumed.
- **CB-4.1 chose honest UX over aspirational** — removed the "it's now a node on your map" success line (false until CB-5b); `Adjust`/`Discard` now actually dismiss the card.

## Specialist Flags (gotchas git diff won't show)
- **Dev-run staleness cost real time today.** A running `artemis serve` (uvicorn) and `tauri dev` client do NOT auto-reload on new commits — both were silently running *yesterday's* code (brain returned 404 for `/app/capabilities`; client lacked the CB-3 gateway commands), which read as "build mode is broken" when it wasn't. **Restart both from HEAD before any live test.** Worth adding `uvicorn --reload` to the dev brain launch and making the launcher always run current source.
- **Codex's sandbox cannot start vitest** — it walks up to read `vite.config.ts`/`../..` and hits "Access is denied." Codex can run `npm run typecheck`/`lint` and `cargo test`/`clippy` fine, but the **host must run `npm run test` (vitest)** for every client spec. (Same for plain `uv run` when a brain process locks the venv exe — use `uv run --no-sync`.)
- **The live run caught what hermetic vitest didn't** (reaffirms the live-smoke rule): dead `Adjust`/`Discard` buttons and a false success message — both passed unit tests (which checked state transitions, not UX completeness). Always live-run a UI slice.
- **The guard's trigger depends on what the model authors, not the prompt.** Asked for an "email extractor," the model wrote a *network-free* local `.eml`/mbox parser → guard passed → it built (`email-extractor-store` is now in `capabilities/library/`). Asking for "read my email over IMAP" makes it author `imaplib`/`ssl` → guard blocks. So "email" alone isn't blocked; *network reach* is.

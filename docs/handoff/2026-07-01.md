# Handoff — 2026-07-01: capability-build slice (build-by-chat) — built + proven live

The dogfood loop now works in the real app: type a build request in the Ask popup → gate the plan → watch the real sandbox verify → gate the result → it's installed. Specs `CB-1, CB-2, CB-3, CB-4, CB-4.1, CB-5a` all in `done/`; details + commits are in `docs/status.md` and `git log`. This handoff is the narrative git/status can't reconstruct.

## Decisions Made
- **The network guard is an AST import-level scan, NOT a security boundary** — locked as the slice's load-bearing contract (`scan_for_unsafe_imports`, written into the CB-1 spec). It enforces the *product* rule "network/secret capabilities wait for WSL2," and keeps honest capabilities honest; a determined model could evade it (`os.system`/`getattr`/`eval`). **WSL2 isolation remains the real boundary** before any network-touching capability runs. Do not let anyone treat the scan as containment.
- **First dogfood capability is network-free by choice** (owner). The engine already proved it can author IMAP-class code (the 2026-06-30 `email_intake` forge test); the chat layer doesn't need a network capability to prove itself, so we kept WSL2 + the secret-paste gate out of this slice.
- **Engine-first sequencing, client-against-built-brain.** Brain (CB-1/CB-2) first, then the gateway (CB-3), then UI (CB-4) — the same pattern as the CR-* revival. CB-3/CB-4 specs were frozen only *after* CB-2's wire shape was built + live-smoked, which avoided churn.
- **Build-stream wire = named SSE events** (`event: status` / `event: result` / `[DONE]`), deliberately distinct from `/app/ask`'s bare-`data:` text stream; CB-3 added a separate `parse_build_frame` rather than overloading the ask parser. In-flight proposals live in an **in-memory** dict on `app.state` (not durable — a build straddling a brain restart is lost; acceptable for a single-user hub).
- **CB-5 split: data (CB-5a, built) vs node visual (CB-5b, deferred).** The map-node visual is blocked on an owner design decision ("what a capability node looks like"). **Owner directive: fan out display-research agents before speccing CB-5b** (parallel research pre-authorized). CB-5a deliberately ships *only* the brain `/app/capabilities` list — the Tauri list command + TS wrapper were left for CB-5b as dead-until-consumed.
- **CB-4.1 chose honest UX over aspirational** — removed the "it's now a node on your map" success line (false until CB-5b); `Adjust`/`Discard` now actually dismiss the card.

# Handoff — 2026-07-01 (later): `enabler-wsl2-spike` — built, 7/7 checks pass unprivileged

Built enabler #1 of ADR-035 (WSL2 isolation PoC, `poc/wsl2_sandbox/`). Codex authored the three
files from the spec; the Opus host provisioned WSL, ran the 7 acceptance checks against live WSL,
fixed check 4, and recorded findings. All 7 pass **unprivileged**. Spec in `done/`; results table +
full findings in `poc/wsl2_sandbox/README.md`. This section is what git/status can't reconstruct.

## Decisions Made
- **`unshare --net` is now conditional, not always-on (deviation from the spec).** The spec's two
  mechanisms are mutually exclusive: a fully-isolated `--net` netns has no route out, so the DNS-
  sinkhole *allowlist* case couldn't reach even the permitted domain (check 4 failed at name
  resolution). Fix in `isolate.sh`: empty allowlist = pure no-network (`--net`); non-empty allowlist =
  host connectivity + a private (mount-ns) `resolv.conf` → sinkhole dnsmasq. This is the mechanism the
  spec described in prose; only the literal "always `--net`" changed. SMALL fork, one file, no new deps.

## Specialist Flags (gotchas git diff won't show)
- **⚠️ (security) The DNS-sinkhole allowlist is a soft, IP-bypassable gate.** It restricts *DNS only* —
  untrusted code that dials a hard-coded IP or ships its own resolver/DoH walks through it. Fine to
  prove the mechanism in a spike; **not** a boundary to trust for real untrusted capabilities.
  **Enabler #2's `FetchSandbox` must add a hard egress boundary:** `iptables` IP-allowlist (resolve
  allowed domains → allow only those IPs, default-drop) or a transparent proxy (`tinyproxy` +
  `iptables … REDIRECT`). The spec deferred the proxy out of the spike; it's now a required #2 input.
- **⚠️ (ops) Unprivileged cgroup caps are silently unenforced.** `systemd-run --scope -p MemoryMax
  -p CPUQuota` *succeeds* unprivileged but the limits are ignored (memory controller not delegated —
  verified: 1 GB alloc under `MemoryMax=256M` succeeded). What actually bounded the `hog` scenario is
  the `ulimit -t 30` CPU-time cap (exit 137 at ~31 s), not the memory cgroup. Enabler #2 needs a
  deliberate choice: root-invoked `systemd-run`, systemd cgroup **delegation** to the run user, or
  `ulimit -v` as the memory floor.

## Blocked / Needs-attention
- none — spike complete, deliverable met.

## What's Next (discoveries)
- **Isolation is fully unprivileged on this host** (netns/pid-ns/mount-ns/tmpfs/CPU-time cap). Enabler
  #2 does NOT need a setuid/root wrapper for *isolation* — root (or cgroup delegation) is only needed
  for enforced **memory/CPU cgroup caps** or an `iptables` egress boundary. That narrows the #2
  privilege decision to exactly those two capabilities.
- **Next build step = enabler #2: the real `FetchSandbox` + `SandboxRunner`** — needs a planning spec.
  The two ⚠️ findings above are its load-bearing design inputs.

# Handoff — 2026-07-01 (session 3): enabler #2 specced + BUILT + proven live; reach-out arc started

One long session: specced enabler #2 (3 specs + ADR-036), **built all three + proved them live on WSL2**,
then kicked off the reach-out-router planning (R1 ready). Commits `5c5c601`, `6bffe93`, `6261402` (+ chore
`c3d0116`); R1 spec + status uncommitted-at-write → committed at pause. Everything reproducible from
git + status.md + `done/`. This is the narrative those can't reconstruct.

## Decisions Made
- **Hardened sandbox = root-privileged *setup*, de-privileged *capability* (ADR-036).** The untrusted
  code runs under a dedicated non-root UID (caps dropped) inside a root-built cage — because the iptables
  `--uid-owner` egress rules only hold if the capability can't `iptables -F`. This reversed the spike's
  unprivileged lean. Egress = **nginx `ssl_preread` SNI allowlist over veth+NAT**, not tinyproxy (can't
  SNI-filter transparent HTTPS) and not TPROXY (absent in the WSL2 kernel).
- **Duck-typed `hardened` flag over `isinstance`** for the guard-relax — deliberately, so the planned
  Lima/macOS runner (ADR-035 #7) also satisfies it; the sandbox object is app-wired, not attacker-controlled.
- **Reach-out: build fresh, v1 as reference — do NOT recover v1 files** (owner call). Saved as memory
  `reachout-build-fresh-not-recover`. Only the egress SSRF guard is lifted near-verbatim. R1 pins the
  validated IP (closes DNS-rebinding), which the v1 `done/DR-b-web-access.md` left as a silent residual.
- **Reach-out-router scope = full shared Telegram+Ask ingress** (owner chose end-state over Ask-only),
  so R4 must build the inbound `TransportPort.receive()` dispatch loop that doesn't exist today.
- **Web-search provider = Tavily default** (env key stopgap), Brave a future protocol swap.

## Specialist Flags (what the reviews/live-smoke surfaced that git won't show)
- **Live-smoke is non-negotiable for the sandbox.** Codex built the runner unit-green, but real WSL2 exposed
  **5 integration bugs mocks can't catch**: `wsl.exe wslpath` mangles backslashes (→ pure-Python convert);
  nginx doesn't load its dynamic `stream` module under a standalone `-c`; **`ip netns exec` remounts `/sys`
  and shadows `/sys/fs/cgroup`** (the cgroup-join silently failed → two tests were FALSE-passing); WSL's
  `/etc/resolv.conf` is a dangling symlink (bind fails); and the netns had **no upstream at all** (missing
  veth+NAT) so the proxy couldn't egress. Also: WSL2 cgroup `memory.max` doesn't reliably enforce even as
  root → **`ulimit -v` is the actual memory enforcer** (cgroup kept as defense-in-depth).
- **Security re-review earns its keep on security specs.** Each dispatched review round found *deeper* BLOCKs
  than the last (policy-wiring: IP-literal SSRF only caught on round 2; runner: newline-injection into the
  nginx config on round 2). The R1 web-primitives review caught 3 real SSRF BLOCKs incl. DNS-rebinding.
  Budget for ≥2 review rounds on any egress/guard-touching spec.
- **A prior `done/DR-b-web-access.md` spec exists for the web layer** — it's the authoritative reference for
  correct details (`tldextract.registered_domain`, `follow_redirects=False`, `pip-audit` gate). Consult it
  when building R1/R2, not just archive/v1 source.

## Blocked Actions
- none. (Provisioning the WSL2 box for the live smoke — nginx/libnginx-mod-stream/dnsmasq/`artemis-cap` UID,
  host nginx/dnsmasq disabled — was done ad hoc this session; it is NOT captured as a repeatable step. The

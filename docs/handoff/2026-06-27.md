## Decisions Made

- **m2-win-b Task 1 — interop mechanism (GRAY fork, owner-resolved).** winsdk 1.0.0b10
  exposes **no** `IUserConsentVerifierInterop` / `RequestVerificationForWindowAsync` at
  the Python level (only the bare UWP `request_verification_async`, which doesn't reliably
  prompt for a console process). Surfaced to owner → owner chose the **ctypes COM shim**.
  Implemented `verify()` as a raw COM bridge; the parameterized
  `IAsyncOperation<UserConsentVerificationResult>` IID is derived in-code via the WinRT
  pinterface SHA-1 algorithm (unit-checked against the published `IReference<bool>` IID).
  **Only validatable by a real gesture** — see What's Next.
- **m2-win-b — `unlock()` maps `NoConsoleWindowError → UnlockUnavailableError`** so the
  no-console-HWND path is "unavailable" per Assumption #2 (the literal Task 2 snippet would
  have let it escape as a raw `HelloError`). In-file refinement.
- **m2-win-b — added public `WindowsKeyProvider.unlocked_scope_count`** so the CLI reports
  the scope count without poking private attributes. Minor in-file addition.
- **finance-fail-closed built inline (Opus host), not Codex** — trivial deterministic
  2-line edit; Codex subprocess overhead/timeout risk not worth it.
- **m2-win-b built inline (Opus host)** — the COM shim is security-critical and unverifiable

## Specialist Flags

- **Cross-model (Codex) review earned its keep on the COM path.** It caught a **real
  production-crash bug**: `hello_available()` called `asyncio.run()`, but `unlock()` runs
  inside the async FastAPI `lifespan` loop → `RuntimeError`. My unit/lifespan tests *hid* it
  by mocking `hello_available`. Fixed by offloading the winsdk coroutine to a worker thread;
  added a regression test that calls `hello_available()` from inside an event loop. It also
  surfaced a latent **64-bit HWND truncation** (default-prototyped `GetConsoleWindow` returns
  `c_int`) — fixed by declaring explicit `argtypes`/`restype` for the win32/`combase` calls.
  Round-2 re-review CLEAN (its one round-2 FLAG — "GetResults at wrong vtable slot" — was a
  **false positive**: `GetResults` is correctly slot 8 on the `IAsyncOperation<T>` pointer
  [QI/AddRef/Release 0-2, IInspectable 3-5, put_Completed 6, get_Completed 7, GetResults 8];
  Codex confused it with an `IAsyncInfo` slot).
- **`uv sync --reinstall-package artemis` silently dropped the `agentic` optional group**
  (uninstalled `openhands`), which red-failed one test on the first full run. Restored with
  `uv sync --group agentic`. Reminder: host-verify needs the agentic group synced (known
  gotcha) — and a bare `uv sync` re-prunes it.

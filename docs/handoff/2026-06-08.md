## Decisions Made (single-writer reconciliations — the WHY)

1. **M2-a volume-mount → appended as Task 10, NOT inserted-at-Task-6-with-renumber.** The manifest said
   "insert between Task 5 and Task 6; renumber consistently," and its pre-check's stated goal was "keep
   cross-refs to 'Task 9' resolving." The gated Secure-Enclave task is Task 9 and is referenced by name from
   M2-a internals **and** externally (M2-c edit 5, M2-d). Renumbering would shift gated→Task 10 and break
   those cross-file refs. Appending the volume task as **Task 10** keeps gated = Task 9 with **zero** cross-ref
   churn — which is exactly the pre-check's true intent. Deviation from literal placement was deliberate.
   → If anyone re-touches M2-a tasks: gated SE+volume work is Task 9; the volume-mount struct is Task 10.

2. **M1-a sensitivity flag → reuse existing `data_scope`, did NOT add a `sensitive: bool`.** The
   `ModuleManifest` already carries `data_scope: DataScope` (`owner-private | guest-visible | shared`) — a
   strict superset of a sensitive boolean. Adding a second field = two sources of truth (a smell). So the
   README cross-milestone dep ("M5-c reads `sensitive: bool`/`data_scope`; if absent add `sensitive`") was
   resolved as **`data_scope` present → no new field**. **Consequence that rippled into M5-c:** M5-c's
   `tier_for` was rewritten from a hardcoded name-keyed set (`{"finance","health",...}`) to
   `tier_for(data_scope: DataScope) -> Tier` where `sensitive ≡ data_scope == OWNER_PRIVATE`. This made
   **M1-a a hard prerequisite of M5-c** (tier.py imports `DataScope`; the Gateway resolves the matched
   module's `data_scope` via the registry through `Brain.pre_route`). The manifest's drafted name-keyed
   stopgap was deliberately NOT used.

3. **M1 cross-milestone back-fills added proactively** so M5 consumers resolve at build time:
   `Brain.respond_stream(text, scope) -> AsyncIterator[str]` (M1-b, consumed by M5-d) and
   `Brain.pre_route(text, scope) -> str | None` (M1-b + surfaced on the M1-c Gateway, consumed by M5-c).
   Both are real new methods + tests in M1-b/M1-c — not just notes. They're the load-bearing edges that make
   M5-c's Tier-pre-serve gate and M5-d's sentence-streaming possible.

4. **M4 internal cross-refs reconciled (single writer):** M4-a's `add`/`update` gained the A-MEM kwargs
   (`keywords`/`contextual_description`/`linked_ids`) — the M4-a manifest only added the *columns*, but M4-b
   edit 7 + the README dep require M4-a to *accept* them, so I added the kwargs. `compute_fact_key` gained
   `object_` (cardinality-aware). `bump_access`/`purge` live in **M4-a's** repository; M4-c only *calls* them
   (its manifest stripped repository.py from M4-c's Modify/git-add lists) — so there is exactly one
   hard-delete primitive (`repo.purge`) in the codebase, in M4-a.

5. **M5-c voiceprint store — hardened beyond a pure marker resolution.** The draft's fallback ("if the
   Tier-0 read path isn't ready, store plaintext under proactive/ with a FLAG") was replaced with **BLOCK on
   M2-c rather than store plaintext** — plaintext voiceprints at rest are not acceptable; embeddings-only,
   Tier-0-key-encrypted, never-logged is the requirement. This is a security tightening, per the manifest edit 6.

## Specialist Flags (non-obvious gotchas a diff won't show)

- **M3-a LanceDB path is deliberately loose.** `volume_vectors_dir` is drafted as
  `<volume_root>/<scope>/lancedb` per the manifest, while M2-a actually mounts the encrypted volume at
  `/opt/artemis/<slot>/<scope>/vault/`. The exact `…/vault/lancedb` path reconciliation was **deferred to the
  on-hardware integration (M3-a Task 8)** by design (manifest: "rename at integration; one-line adapter
  change"). Off-hardware `volume_root` defaults to `data_root`, so tests don't care. A builder should expect a
  one-line path adjustment when the broker-mounted volume is live.
- **M4-b A.U.D.N. cardinality routing:** SINGLE relations use mem0's UPDATE rubric; MULTI relations NEVER
  take an in-place UPDATE (a different object = a coexisting ADD, or tombstone+ADD on explicit supersession).
  The decider gets the cardinality via `repo.cardinality_of()` before deciding. Easy to get wrong if read fast.
- **M2-d is a real BLOCK gate**, not advisory: M3-* and M4-* must not begin until its verdict is
  PASS/CONDITIONAL-PASS. Encoded in M2-d's acceptance + as a prereq the M3/M4 specs reference.

## Specialist Flags
- **(security/robustness) `QuarantinedReader.read()` crashes the batch on a model transport error.**
  The `model.complete()` call is *outside* the try/except that handles parse failures, so an Ollama
  `400` (or any HTTP/timeout error) on a single untrusted email kills the whole poll. The untrusted-
  input reader should degrade-and-continue on transport errors exactly like it does on JSON-parse
  errors. `untrusted/quarantine.py:132-140` vs the `except` at `:143`. (BACKLOG, 2026-06-28.)
- **(ai-systems) Ollama ignores the OpenAI `response_format: json_schema`** — `qwen3:4b` returned
  prose for every quarantine read (7/7 → `JSONDecodeError` degrade). Needs Ollama's native `format`
  param (or prompt-coax / different reader). The `400` was likely `num_ctx` overflow on a long email.
  Exactly the tuning signal the observe harness exists to surface. (BACKLOG, 2026-06-28.)

- **`artemis-google-auth login` registers no scopes → bails** (`"no scopes…"`). `cli.py login`
  computes `required_scopes()` but imports no connector modules. Root cause: inconsistent
  registration — `modules/calendar` registers at module import (top-level), but `modules/gmail`
  registers only inside `build_gmail_manifest()` (`register_gmail_scope()`). The standalone CLI never
  composes manifests. Fix: have the CLI trigger connector scope registration before login (and/or move
  gmail's registration to module top-level for parity). Runbook's "read-only" claim is also stale —
  calendar registers `calendar.events` (write). (BACKLOG, 2026-06-28.)
